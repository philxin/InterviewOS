# InterviewOS V2 执行看板

## 1. 文档目标

用于记录 V2 当前阶段的执行状态、交付物与阻塞项，按任务流推进，不按天拆分。

V2 范围：

- 用户系统与数据隔离
- 训练方向初始化（轻量岗位选择）
- 批量知识导入与简单标签体系
- 更真实的题目结构
- 更可信的评分反馈结构
- 前后端 V2 主链路联调

V2 明确不做：

- 简历深度解析
- 复杂岗位画像
- 多轮对话
- 复杂图表统计
- 社交 / 排行榜
- 过多文件格式导入
- 超复杂标签系统
- 双数据库拆分
- 复杂权限体系
- 语义检索 / RAG

---

## 2. 看板规则

状态定义：

- `TODO`：未开始
- `DOING`：进行中（最多 1-2 个并行卡片）
- `BLOCKED`：被外部依赖阻塞
- `DONE`：完成并通过当前验收标准

优先级定义：

- `P0`：阻塞 V2 主路径
- `P1`：重要但不阻塞 V2 主路径
- `P2`：增强项 / 预留项

完成标准（DoD）：

- 代码已落库或文档已冻结
- 主路径自测通过
- 错误路径有可预期返回
- 接口 / 数据 / 页面变更已同步文档

## 5. 任务依赖关系（执行顺序）

P0 主链路：

`V2-01 -> V2-02 -> V2-03 -> V2-04 -> V2-05 -> V2-06 -> V2-07 -> V2-08 -> V2-09 -> V2-10`

并行支线：

- `V2-07` 可在 `V2-01` 基本稳定后并行
- `V2-11`、`V2-12`、`V2-13`、`V2-14` 在 `V2-10` 前不应抢占主线
- `V2-15`、`V2-16`、`V2-17` 仅在 P0 稳定后进入排期

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 范围膨胀，把 P2 提前做到 P0 | 主路径延期 | 严格以 `V2-01 ~ V2-10` 作为首批上线范围 |
| 评分结构改版后仍然“不可信” | 结果页价值下降 | 先冻结反馈结构，再单独验收 Prompt 输出 |
| 训练模型新旧并存导致代码混乱 | 维护成本高 | 明确 V2 主链路只走 `TrainingSession/TrainingQuestion` |
| Security 改造后前端大面积 401 | 联调效率下降 | 先完成 `/auth/me` 和登录守卫，再逐步切受保护接口 |
| 批量导入引入脏数据 | 知识点质量下降 | 做逐项校验与失败项回传，禁止静默跳过 |
