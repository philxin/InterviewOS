# InterviewOS V2 后端实现计划

## 📁 目标

V2 后端实现的目标不是推翻 V1，而是在现有 Spring Boot 结构上完成以下升级：

1. 引入用户鉴权与数据隔离
2. 将训练流程从 V1 的 `start/submit/history` 升级为 V2 的 `session-based` 模型
3. 重构知识点模型，支持标签、批量导入和来源类型
4. 重构反馈结果结构，使评分结果更可解释
5. 为 P1 / P2 的提示、Dashboard、推荐训练预留扩展空间

---

## 🧭 实现原则

### 1. 最小侵入但不强行兼容旧模型

- 保留已有 `common/`、日志、异常处理、LLM 抽象
- 不强行在 `TrainingRecord` 上继续堆 V2 需求
- 新训练主模型直接采用 `training_session + training_question`

### 2. 鉴权与数据边界必须先立住

- 所有资源默认要求登录
- 用户 owner 校验必须在 Security / Service 层统一处理
- 禁止在 Controller 手动判断 token 或 userId

### 3. 先完成骨架，再补增强能力

- P0 先打通：用户 → 知识点 → 会话训练 → 结果反馈
- P1 再补：提示、文件导入、Dashboard
- P2 再补：推荐训练、自动组题、回练

### 4. 保持现有工程习惯

- 继续使用 Spring Boot 3.3.6 + JPA + Validation + Security
- 不新增不必要框架
- 关键 DTO、Service 公共方法、复杂查询必须补简洁注释

## 🔐 模块一：认证与用户体系

### 新增实体

#### `AppUser`

字段参考 [V2数据库设计.md](./V2数据库设计.md)：

- `id`
- `email`
- `passwordHash`
- `displayName`
- `targetRole`
- `createdAt`
- `updatedAt`

### 新增 Repository

- `AppUserRepository`
  - `Optional<AppUser> findByEmail(String email)`
  - `boolean existsByEmail(String email)`

### 新增 Service

#### `AuthService`

职责：

- 注册
- 登录
- 获取当前用户信息
- 密码校验

#### `CurrentUserService`

职责：

- 从 SecurityContext 获取当前用户
- 提供 `getCurrentUserId()`、`getCurrentUser()` 等统一入口

### 新增 Controller

#### `AuthController`

接口：

- `POST /auth/register`
- `POST /auth/login`
- `GET /auth/me`

认证响应统一返回：

- `token`
- `tokenType = Bearer`
- `expiresIn`
- `user`

### 安全实现建议

1. 引入 JWT 相关依赖
2. 新增 `JwtTokenService`
3. 新增 `JwtAuthenticationFilter`
4. 在 `SecurityConfig` 中：
   - 放行 `/auth/**`
   - 放行 `OPTIONS /**`
   - 其他路径默认认证

### 注意事项

- 密码必须使用 `PasswordEncoder`
- 不在日志中打印密码、token
- owner 维度判断不要放在 Controller

## 📚 模块二：知识点与标签体系

### 改造实体

#### `Knowledge`

V2 字段：

- `id`
- `user`
- `title`
- `content`
- `mastery`
- `sourceType`
- `createdAt`
- `updatedAt`

#### `KnowledgeTag`

字段：

- `id`
- `knowledge`
- `tag`
- `createdAt`

### Repository

#### `KnowledgeRepository`

建议方法：

- `List<Knowledge> findByUserIdOrderByCreatedAtDesc(Long userId)`
- `Optional<Knowledge> findByIdAndUserId(Long id, Long userId)`
- `List<Knowledge> findByUserIdAndMasteryLessThanOrderByMasteryAscUpdatedAtDesc(Long userId, Integer mastery)`
- 按关键字和标签过滤建议使用 `@Query` 或 Specification

#### `KnowledgeTagRepository`

建议方法：

- `List<KnowledgeTag> findByKnowledgeId(Long knowledgeId)`
- `List<String> findDistinctTagsByUserId(Long userId)` 可用自定义查询

### Service

#### `KnowledgeService`

V2 职责：

- 获取当前用户知识点列表
- 获取单个知识点详情
- 创建 / 更新 / 删除知识点
- 查询标签列表
- owner 校验

#### `KnowledgeImportService`

V2 职责：

- 批量导入知识点
- 文件导入任务创建与状态查询（P1）
- 失败项汇总

### Controller

#### `KnowledgeController`

保留并升级：

- `GET /knowledge`
- `GET /knowledge/{id}`
- `POST /knowledge`
- `PUT /knowledge/{id}`
- `DELETE /knowledge/{id}`
- `POST /knowledge/batch-import`
- `GET /knowledge/tags`

P1 预留：

- `POST /knowledge/file-imports`
- `GET /knowledge/file-imports/{importId}`

### 关键实现点

1. `KnowledgeResponse` 需要增加：
   - `tags`
   - `sourceType`
   - `status`
   - `updatedAt`
2. 标签写入时统一清洗大小写与空格
3. 批量导入要返回 `createdCount + failedItems`
4. V2 中知识点删除语义为归档，不做物理删除

## 🎯 模块三：训练会话与反馈重构

### 替换旧模型

V1：

- `TrainingRecord`
- `TrainingRecordRepository`
- `TrainingController`
- `StartTrainingRequest/Response`
- `SubmitTrainingRequest/Response`

V2：

- `TrainingSession`
- `TrainingQuestion`
- `TrainingSessionRepository`
- `TrainingQuestionRepository`
- `TrainingSessionController`

### 新增实体

#### `TrainingSession`

职责：

- 表示一次训练入口
- 管理状态、题型、难度、提示开关
- 保存 session 级摘要字段：`summaryScore / summaryBand / summaryMajorIssue`

#### `TrainingQuestion`

职责：

- 保存问题正文
- 保存用户回答
- 保存反馈结构
- 保存掌握度变化

### Repository

#### `TrainingSessionRepository`

建议方法：

- `Optional<TrainingSession> findByIdAndUserId(UUID id, Long userId)`
- `List<TrainingSession> findByUserIdOrderByCreatedAtDesc(Long userId)`

#### `TrainingQuestionRepository`

建议方法：

- `List<TrainingQuestion> findBySessionIdOrderByOrderNoAsc(UUID sessionId)`
- `List<TrainingQuestion> findByKnowledgeIdOrderByCreatedAtDesc(Long knowledgeId)`

### Service

#### `TrainingSessionService`

P0 核心方法建议：

```java
public TrainingSessionStartResponse startSession(StartTrainingSessionCommand command);
public HintResponse getHint(UUID sessionId, UUID questionId);
public FeedbackResponse submitAnswer(UUID sessionId, SubmitAnswerCommand command);
public List<TrainingSessionSummary> getSessionHistory(Long knowledgeId);
public TrainingSessionDetail getSessionDetail(UUID sessionId);
```

#### `MasteryService`

职责：

- 统一掌握度更新公式
- 统一薄弱标签识别逻辑
- 避免分散在训练服务中写魔法值

### 反馈结构改造

V1 结构：

- `accuracy`
- `depth`
- `clarity`
- `overall`
- `strengths`
- `weaknesses`
- `suggestions`
- `exampleAnswer`

V2 结构：

- `score`
- `band`
- `majorIssue`
- `missingPoints`
- `betterAnswerApproach`
- `naturalExampleAnswer`
- `weakTags`
- `masteryBefore`
- `masteryAfter`

### Controller

#### `TrainingSessionController`

接口：

- `POST /training/sessions`
- `POST /training/sessions/{sessionId}/questions/{questionId}/hint`
- `POST /training/sessions/{sessionId}/answers`
- `GET /training/sessions`
- `GET /training/sessions/{sessionId}`

### 关键实现点

1. P0 默认一个 session 一道题，但结构上允许未来扩展到 3-5 题
2. 轻量追问可通过 `parentQuestionId` 支持，但不做完整多轮对话
3. `score` 与 `band` 要同步保存，避免每次查询时二次推导
4. session 历史摘要由服务端聚合生成，不能交给前端临时拼装

## 🤖 模块四：LLM 能力重构

### 保留

- `LLMService`
- `LLMServiceImpl`
- `PromptTemplate`

### 重构方向

V1 的 `LLMService` 偏向：

- `generateQuestion(title, content)`
- `evaluateAnswer(question, answer)`

V2 建议升级为更强语义输入：

```java
QuestionGenerationResult generateQuestion(QuestionGenerationRequest request);
FeedbackGenerationResult evaluateAnswer(FeedbackGenerationRequest request);
```

### 新增 DTO

#### `QuestionGenerationRequest`

- `knowledgeTitle`
- `knowledgeContent`
- `targetRole`
- `questionType`
- `difficulty`
- `previousQuestionContext`（可选）

#### `FeedbackGenerationRequest`

- `questionText`
- `questionType`
- `difficulty`
- `userAnswer`
- `knowledgeTitle`
- `knowledgeContent`
- `hintUsed`

### Prompt 重构重点

#### 出题 Prompt

必须显式约束：

- 输出题型
- 输出题目正文
- 避免重复模板问法
- 更贴近面试场景

#### 评分 Prompt

必须显式约束：

- 输出 `score`
- 输出 `band`
- 输出简短 `majorIssue`
- 输出 `missingPoints`
- 输出 `betterAnswerApproach`
- 输出更自然的示范表达

### 日志要求

- 继续只记录长度、耗时、指纹
- 不记录完整 prompt / response

## ⚠️ 风险点

### 风险 1：旧训练模型与新训练模型并存混乱

- 处理方式：明确 `TrainingRecord` 为旧模型，V2 主链路只走新模型

### 风险 2：Security 改造导致前端全链路 401

- 处理方式：先落 `auth/me` 与最小登录流程，再切知识点接口保护

### 风险 3：LLM 输出结构漂移

- 处理方式：使用更强约束 Prompt + 结果解析兜底 + 上游异常分类

### 风险 4：批量导入带来脏数据

- 处理方式：做逐项校验与失败项回传，不整批静默吞错

### 风险 5：P1/P2 范围前置

- 处理方式：代码结构可预留，但任务排期必须先收敛到 P0

## ✅ 完成标志

当满足以下条件时，V2 后端主骨架可视为完成：

1. 认证体系可用，资源完成用户隔离
2. 知识点与标签模型可用，支持批量导入
3. 新训练会话接口可完成出题、回答、反馈、历史查询
4. 反馈结构已切换为 V2 契约
5. 旧 V1 模型不再作为主链路对外暴露

后续实现阶段应与 [V2版本规划.md](./V2版本规划.md)、[API接口设计.md](./API接口设计.md)、[V2数据库设计.md](./V2数据库设计.md) 保持一致。
