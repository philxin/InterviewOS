# InterviewOS V3 数据库设计

## 📊 概述

V3 数据库设计不是把向量列硬塞进 `knowledge` 表，而是为以下能力提供稳定数据基础：

1. 文件导入后的文档级元数据管理
2. chunk 切片与向量索引
3. 训练链路中的引用快照与可追溯校准
4. 自动知识点识别的后续扩展
5. 模型版本切换、重建索引、失败重试的工程化能力

V3 继续以 **PostgreSQL 单库 + pgvector** 为主落地路径，不引入独立向量数据库。

---

## 🧭 设计原则

### 1. 用户隔离优先

- 所有文档、chunk、引用快照必须归属于具体用户
- 检索查询默认带 `user_id`
- 禁止只按 chunk 主键直接读取跨用户数据

### 2. 训练主数据与检索索引解耦

- `knowledge` 继续服务于训练主模型与掌握度体系
- 文档与向量索引单独落表
- 不把 `knowledge` 直接当作向量索引载体

### 3. 幂等与可重建优先

- 导入任务支持安全重试
- chunk 与 embedding 支持按内容 hash 去重或重建
- 模型切换时允许双表 / 新索引迁移，而不是原地覆盖

### 4. 可追溯优先

- 训练题目与反馈使用过的引用片段必须可回看
- 即使原文后续重建，也应保留训练时的引用快照

### 5. 先支持 V3 P0，再预留 P1

- P0 先完成文档、chunk、引用快照与任务进度
- P1 再增加自动知识点候选表

---

## 🔄 V3 相比 V2 的核心变化

### V2 主体

- `app_user`
- `knowledge`
- `knowledge_tag`
- `training_session`
- `training_question`
- `knowledge_file_import`

### V3 新增 / 变更

- `knowledge_file_import`（扩展进度与索引字段）
- `knowledge_document`（新增）
- `knowledge_chunk`（新增）
- `training_question_reference`（新增）
- `knowledge_concept`（P1 可选）
- `rag_trace`（P1 可选）

### 变化说明

1. 文件导入不再直接等价于“创建 1 条知识点”
2. 文档与 chunk 成为检索主实体
3. 训练过程需要持久化引用快照，保证反馈可复现
4. 自动知识点识别不直接写回 `knowledge`，而是先形成候选

---

## 1. `knowledge_file_import`（V3 扩展）

V2 已有导入任务表，V3 继续复用，但补齐进度与索引信息。

### 新增字段

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `document_id` | `uuid` | NULL, FK | 对应生成的文档主键 |
| `content_hash` | `varchar(64)` | NULL | 原始文本摘要，用于幂等与排查 |
| `total_chunks` | `integer` | NOT NULL, DEFAULT 0 | 预计 / 实际切片数量 |
| `embedded_chunks` | `integer` | NOT NULL, DEFAULT 0 | 已成功向量化数量 |
| `failed_chunks` | `integer` | NOT NULL, DEFAULT 0 | 向量化失败数量 |
| `parser_version` | `varchar(50)` | NULL | 解析器版本 |
| `embedding_model` | `varchar(100)` | NULL | 使用的 embedding 模型名 |
| `embedding_dim` | `integer` | NULL | embedding 维度 |

### 状态值建议

- `PENDING`
- `PROCESSING`
- `CHUNKING`
- `EMBEDDING`
- `SUCCESS`
- `PARTIAL`
- `FAILED`

### 索引建议

- 保留：`idx_file_import_user_created`
- 保留：`idx_file_import_status_created`
- 新增：`idx_file_import_user_status_created`：`(user_id, status, created_at desc)`

### 业务规则

- `created_count` 保留用于兼容旧接口，但不再作为 V3 核心进度字段
- 同一个 `import_id` 不得重复创建第二份 document
- 失败重试时只补齐缺失 chunk，不重复写入已成功 chunk

## 2. `knowledge_document`

存储上传文件在系统中的文档级元信息。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `uuid` | PK | 文档主键 |
| `user_id` | `bigint` | NOT NULL, FK | 所属用户 |
| `import_id` | `uuid` | NOT NULL, UNIQUE, FK | 来源导入任务 |
| `title` | `varchar(200)` | NOT NULL | 文档标题，默认取文件名 |
| `original_file_name` | `varchar(255)` | NOT NULL | 原始文件名 |
| `content_type` | `varchar(100)` | NOT NULL | MIME 类型 |
| `content_hash` | `varchar(64)` | NOT NULL | 文本摘要 |
| `parser_version` | `varchar(50)` | NULL | 解析器版本 |
| `chunk_strategy` | `varchar(50)` | NOT NULL | 切片策略标识 |
| `embedding_model` | `varchar(100)` | NULL | 当前索引使用模型 |
| `embedding_dim` | `integer` | NULL | 当前索引维度 |
| `status` | `varchar(20)` | NOT NULL | `PROCESSING / ACTIVE / ARCHIVED / FAILED` |
| `total_chunks` | `integer` | NOT NULL, DEFAULT 0 | 总切片数 |
| `active_chunks` | `integer` | NOT NULL, DEFAULT 0 | 当前可检索切片数 |
| `source_meta` | `jsonb` | NULL | 页码、解析警告等来源元信息 |
| `indexed_at` | `timestamp` | NULL | 最近一次索引完成时间 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |
| `updated_at` | `timestamp` | NOT NULL | 更新时间 |
| `archived_at` | `timestamp` | NULL | 归档时间 |

### 外键

- `fk_knowledge_document_user_id`: `user_id -> app_user(id)`
- `fk_knowledge_document_import_id`: `import_id -> knowledge_file_import(id)`

### 索引

- `uk_knowledge_document_import_id`
- `idx_knowledge_document_user_status_updated`：`(user_id, status, updated_at desc)`
- `idx_knowledge_document_user_created`：`(user_id, created_at desc)`
- `idx_knowledge_document_content_hash`：`(user_id, content_hash)`

### 业务规则

- 一个导入任务最多对应一个文档
- 文档归档后，默认不参与检索
- 文档更新内容时，不直接原地改 chunk，而是触发重新切片与重新索引

## 3. `knowledge_chunk`

存储文档切片与向量索引，是 V3 检索的核心事实表。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `bigserial` | PK | 切片主键 |
| `document_id` | `uuid` | NOT NULL, FK | 所属文档 |
| `user_id` | `bigint` | NOT NULL, FK | 所属用户，便于安全过滤 |
| `chunk_index` | `integer` | NOT NULL | 文档内顺序，从 0 开始 |
| `text` | `text` | NOT NULL | 切片正文 |
| `text_hash` | `varchar(64)` | NOT NULL | 切片摘要 |
| `char_count` | `integer` | NOT NULL | 字符数 |
| `token_count` | `integer` | NULL | 估算 token 数 |
| `page_from` | `integer` | NULL | 起始页码 |
| `page_to` | `integer` | NULL | 结束页码 |
| `start_offset` | `integer` | NULL | 文本起始偏移 |
| `end_offset` | `integer` | NULL | 文本结束偏移 |
| `embedding` | `vector` | NULL | pgvector 向量列 |
| `embedding_model` | `varchar(100)` | NULL | 当前向量模型 |
| `embedding_dim` | `integer` | NULL | 当前向量维度 |
| `status` | `varchar(20)` | NOT NULL | `PENDING / READY / FAILED / DELETED` |
| `failure_reason` | `varchar(255)` | NULL | 向量化失败原因摘要 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |
| `updated_at` | `timestamp` | NOT NULL | 更新时间 |
| `deleted_at` | `timestamp` | NULL | 软删除时间 |

### 外键

- `fk_knowledge_chunk_document_id`: `document_id -> knowledge_document(id)`
- `fk_knowledge_chunk_user_id`: `user_id -> app_user(id)`

### 约束与索引

- 唯一约束：`uk_knowledge_chunk_document_index`：`(document_id, chunk_index)`
- 唯一约束：`uk_knowledge_chunk_document_hash`：`(document_id, text_hash)`
- 普通索引：`idx_knowledge_chunk_user_status_document`：`(user_id, status, document_id, chunk_index)`
- 普通索引：`idx_knowledge_chunk_document_status`：`(document_id, status, chunk_index)`
- 向量索引：`ivfflat` 或 `hnsw`，仅对 `status = 'READY'` 的数据使用

### 业务规则

- 检索时必须至少按 `user_id` 和 `status = READY` 过滤
- 不允许跨模型、跨维度混合比较相似度
- 文档重建索引时，旧 chunk 应软删除或失效，而不是直接覆盖

## 4. `training_question_reference`

存储训练题目、提示、反馈使用过的引用快照。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `bigserial` | PK | 主键 |
| `question_id` | `uuid` | NOT NULL, FK | 对应训练题 |
| `chunk_id` | `bigint` | NULL, FK | 引用的原始 chunk |
| `usage_type` | `varchar(20)` | NOT NULL | `QUESTION / HINT / FEEDBACK` |
| `rank_no` | `integer` | NOT NULL | 本次引用排序 |
| `similarity_score` | `double precision` | NULL | 当次检索得分 / 距离转换值 |
| `document_title_snapshot` | `varchar(200)` | NOT NULL | 文档标题快照 |
| `excerpt_snapshot` | `text` | NOT NULL | 引用片段快照 |
| `locator_snapshot` | `jsonb` | NULL | 页码 / 偏移等定位信息快照 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |

### 外键

- `fk_training_question_reference_question_id`: `question_id -> training_question(id)`
- `fk_training_question_reference_chunk_id`: `chunk_id -> knowledge_chunk(id)`

### 索引

- `idx_tqr_question_usage_rank`：`(question_id, usage_type, rank_no)`
- `idx_tqr_chunk_id`：`(chunk_id)`

### 业务规则

- 训练启动时用于出题的引用集合应固定并落库
- 提交答案时用于反馈的引用集合应可回放
- 即使原 chunk 后续失效，也应保留快照文本用于历史展示

## 🔁 迁移策略建议

### 1. 保留 V2 主表，不做破坏性改造

- `knowledge`
- `training_session`
- `training_question`
- `knowledge_file_import`

### 2. 先增量加表，再接新流程

- 先建 `knowledge_document`
- 再建 `knowledge_chunk`
- 再建 `training_question_reference`
- 最后让训练链路切读新表

### 3. 旧数据不强制回填

- V2 旧知识点先继续按原路径训练
- V3 文件导入的新数据按文档 / chunk 路径索引
- 是否为旧 `Knowledge` 建索引，放到后续独立任务评估

### 4. 模型版本迁移采用“并行索引”策略

- 当 embedding 模型变更时，新建新索引或新批次数据
- 检索只读取当前激活模型对应的数据
- 不建议直接原地覆盖旧向量

---

## ✅ V3 数据库完成标准

- 文档、chunk、引用快照三层结构落地
- 检索数据默认具备用户隔离字段
- 导入任务具备阶段进度、部分失败与重试能力
- 训练历史可追溯引用过哪些片段
- 模型版本、维度、重建索引具备基本扩展空间
