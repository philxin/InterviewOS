# InterviewOS V3 交接摘要

## 1. 项目定位

InterviewOS `V3` 的目标是把产品从“基于单条知识点训练”升级为“基于文件级知识库做检索增强训练”的系统。

V3 重点解决：

1. 文件导入后的文档、切片、向量化处理
2. 检索增强训练的出题、hint、feedback 校准
3. 训练引用快照与结果可追溯性
4. 文档列表、检索验证与候选知识点识别

## 2. 当前技术栈

后端：

- Java 21
- Spring Boot 3.3.6
- Spring AI 1.0.0-M5
- Spring Web / Data JPA / Validation / Security / Redis
- PostgreSQL + pgvector

前端：

- Vue 3
- TypeScript
- Vite
- Pinia
- Vue Router
- Tailwind CSS
- Axios
- ECharts
- Element Plus

## 3. V3 已完成内容

### 后端

1. `KnowledgeDocument`、`KnowledgeChunk`、`TrainingQuestionReference` 数据模型
2. 文件导入状态机升级：解析、切片、向量化、部分失败与成功状态
3. `EmbeddingService` 与 `RetrievalService`
4. 文档列表、文档详情、最小检索验证接口
5. 训练题目、hint、feedback 的引用快照返回与持久化
6. 候选知识点识别、接受、忽略与合并能力
7. 检索 trace、跨用户隔离与基础 prompt injection 防护

### 前端

1. 文件导入页阶段化状态展示
2. 文档库与检索验证页
3. 训练作答页、结果页、历史详情页的引用展示
4. 候选知识点列表、确认新建、忽略、合并到现有知识点
5. 保持原有 `AppShell + route page` 的 Web 应用模型

### 文档

1. V3 版本规划、数据库设计、接口设计、开发任务清单已冻结
2. V3 执行看板与联调记录已收口
3. 当前 V3 文档已整体归档到 `docs/archive/v3/`

## 4. V3 核心业务规则

1. 所有文档、chunk、检索与训练引用默认按当前登录用户隔离
2. 训练主链路在 V3 引入 retrieval context，但不改变已有核心接口主路径
3. 训练时使用的引用快照需要持久化，保证结果可回看
4. 候选知识点识别不直接覆盖正式 `Knowledge`
5. 前端仍然是传统页面导航模型，不是桌面工作区模型

## 5. 当前核心接口

认证与用户：

1. `POST /api/auth/register`
2. `POST /api/auth/login`
3. `GET /api/auth/me`
4. `PATCH /api/users/me/onboarding`

知识与文档：

1. `GET /api/knowledge`
2. `POST /api/knowledge`
3. `POST /api/knowledge/file-imports`
4. `GET /api/knowledge/file-imports/{importId}`
5. `GET /api/knowledge/documents`
6. `GET /api/knowledge/documents/{documentId}`
7. `GET /api/knowledge/concepts`
8. `POST /api/knowledge/concepts/{conceptId}/accept`
9. `POST /api/knowledge/concepts/{conceptId}/reject`

训练与检索：

1. `POST /api/training/sessions`
2. `POST /api/training/sessions/{sessionId}/answers`
3. `POST /api/training/sessions/{sessionId}/questions/{questionId}/hint`
4. `GET /api/training/sessions`
5. `GET /api/training/sessions/{sessionId}`
6. `POST /api/rag/search`

默认 Base URL：

```text
http://localhost:8080/api
```

## 6. 当前配置与结构结论

1. V3 的后端契约与数据结构可继续作为下一版本的稳定业务基线
2. 现有 `frontend` 已承载完整业务主链路，但 UI/UX 仍是后台式页面导航模型
3. 如果要做桌面化前端重构，建议新建独立应用目录，而不是在现有 `frontend` 上硬改
4. 新前端应复用 V3 API 契约，避免后端同时承担桌面化重构风险

## 7. 当前安全与可观测性结论

1. 登录态通过 Bearer Token 传递
2. owner 级资源访问控制已进入服务主链路
3. 检索结果按用户隔离，不允许跨用户召回
4. 日志默认脱敏，不打印密码、token 与大段原文正文
5. 检索 trace 与导入状态已具备最小可观测性

## 8. 联调与验收结论

已确认通过：

1. 文件导入主链路
2. 文档列表与检索验证
3. 训练出题、hint、feedback 的引用返回
4. 候选知识点最小闭环
5. 前端构建与后端测试通过

当前结论：

- V3 已完成并归档
- 当前自然下一阶段是 V4：`frontend-desktop` 独立重构、桌面工作区壳层、多窗口与 Dock 交互模型

## 9. 已知边界与后续建议

仍需关注：

1. 现有 `frontend` 信息架构仍是典型页面式应用
2. 业务模块数量已经超过简单顶部导航的舒适范围
3. 训练、文档、历史、邀请等模块缺少统一的工作区语义
4. 如果继续在现有前端上叠加复杂交互，维护成本会明显恶化

后续自然方向：

1. 新建 `frontend-desktop`，保留旧 `frontend` 不动
2. 使用桌面工作区范式承接知识、训练、历史、文档、邀请等模块
3. 将导航模型升级为 Dock、Launchpad、窗口管理与全局命令面板
4. 完成桌面端优先、移动端降级的双壳层策略
