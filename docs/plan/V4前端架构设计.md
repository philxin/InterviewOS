# InterviewOS V4 前端架构设计

## 1. 文档目标

本文档用于冻结 `frontend-desktop` 的前端架构方向，确保 V4 重构不退化成“视觉换皮”。

本文档回答 4 个问题：

1. 新前端为什么要独立目录重写
2. 桌面壳层与窗口系统如何分层
3. 业务模块如何从 page model 迁移为 app model
4. 新旧前端如何并行与灰度

---

## 2. 核心原则

1. **前端独立重写，后端契约复用**
2. **Shell 优先，模块迁移第二**
3. **App Model 优先于 Page Model**
4. **桌面端优先，移动端明确降级**
5. **视觉借鉴 macOS，但不做 1:1 复刻**
6. **P0 先交付稳定窗口系统，不追求复杂度炫技**

---

## 3. 技术选型建议

### 保留

- `Vue 3`
- `TypeScript`
- `Vite`
- `Pinia`
- `Vue Router`
- `Axios`

### 降低依赖或谨慎使用

- `Element Plus`

原因：

- Element Plus 适合管理后台和表单场景
- 桌面工作区的 Dock、Top Bar、Window、Launchpad、Command Palette 更适合自定义组件
- 表单类细分场景可局部复用，但不应主导壳层设计

## 5. 分层设计

### 5.1 App Layer

职责：

- 应用启动
- 全局 provider 注入
- 路由守卫
- 登录态恢复
- 全局错误边界

### 5.2 Desktop Shell Layer

职责：

- 渲染桌面背景
- 渲染 Top Bar 与 Dock
- 承接桌面级交互入口
- 组织窗口管理容器

禁止：

- 直接写业务 API 调用
- 直接承载知识点、训练、历史等具体业务逻辑

### 5.3 Windowing Layer

职责：

- 窗口实例管理
- z-index 与焦点切换
- 最小化、最大化、关闭
- 窗口状态持久化

禁止：

- 业务状态混进 window store
- 每个模块自行维护窗口层级

### 5.4 Module Layer

职责：

- 承载具体业务应用
- 复用 `shared/api` 与 `shared/auth`
- 对外暴露 App 入口组件

禁止：

- 自己绕开 window manager 直接控制壳层
- 自己实现重复的 auth、notification、command palette 逻辑

### 5.5 Shared Layer

职责：

- API client
- Auth token 管理
- 设计 token
- 通用 hooks / utils / base components

## 6. App Model 设计

V4 不再把业务模块视为“路由页面”，而是视为“桌面应用”。

### 首批 App 清单

| App ID | 名称 | 来源模块 | P0 |
|---|---|---|---|
| `knowledge-explorer` | Knowledge Explorer | 知识点 / 文档库 | 是 |
| `training-console` | Training Console | 训练作答 | 是 |
| `history-review` | History Review | 历史与结果 | 是 |
| `launchpad` | Launchpad | 新增 | 否 |
| `notification-center` | Notification Center | 新增 | 否 |
| `workspace-settings` | Workspace Settings | 设置 | 否 |

### App Manifest 建议字段

```ts
interface DesktopAppManifest {
  id: string
  title: string
  icon: string
  route: string
  defaultSize: { width: number; height: number }
  minSize?: { width: number; height: number }
  singleton?: boolean
  mobileMode?: 'page' | 'sheet' | 'disabled'
}
```

说明：

- `singleton`：像设置页、Launchpad 可限制单实例
- `mobileMode`：定义移动端降级方式

## 7. Window Model 设计

### Window State 建议字段

```ts
interface DesktopWindowState {
  id: string
  appId: string
  title: string
  status: 'normal' | 'minimized' | 'maximized'
  isFocused: boolean
  zIndex: number
  position: { x: number; y: number }
  size: { width: number; height: number }
  route?: string
  payload?: Record<string, unknown>
}
```

### P0 窗口能力范围

- 打开
- 关闭
- 聚焦
- 最小化
- 最大化
- 拖拽
- 简单位置记忆

### P0 明确不做

- 任意自由缩放
- 多桌面 Space
- 窗口吸附布局系统
- 复杂 split view

## 8. 路由策略

V4 应采用“深链接 + 窗口状态映射”的混合模式。

规则：

1. 路由仍然存在，用于刷新恢复和分享链接
2. 路由不再直接等价于完整页面布局
3. Desktop Shell 是顶层容器，业务 app 作为其内部窗口存在
4. 登录页、邀请注册页等 guest-only 页面仍保留独立全屏路由

建议：

- guest 路由：`/login`、`/invite/:code`
- desktop 根路由：`/desktop`
- 业务深链接：`/desktop/knowledge`、`/desktop/training/:id`、`/desktop/history`

## 9. 新旧前端并行策略

### 目录策略

- `frontend/`：旧版，冻结维护
- `frontend-desktop/`：新版，独立开发

### 运行策略

- 本地开发：两个前端分开启动
- 构建产物：独立输出
- 文档与 README：分别维护

### 灰度策略建议

1. 默认入口继续指向旧版
2. 提供 `/desktop` 或独立域名作为新版入口
3. P0 达标后，再决定是否切默认入口

## 10. 视觉与交互策略

### 推荐视觉方向

- 风格：Professional Glass
- 关键词：glass, metal, graphite, cool light, spatial depth
- 避免：过度拟物、过亮霓虹、Apple 细节直接复制

### 推荐组件语义

- Top Bar
- Dock
- Window
- App Icon
- Workspace
- Command Palette
- Notification Center

### 交互策略

- 动效必须服务空间感，而不是装饰
- Dock hover、窗口开合、聚焦切换是重点
- 表单控件和业务内容区保持高可读性，不能为了“像系统”牺牲信息密度

## 11. 风险清单

| 风险 | 影响 | 应对 |
|---|---|---|
| 只重写视觉，不重写模型 | 高成本换皮 | 先落 Desktop Shell 与 window store |
| Element Plus 过度介入壳层 | UI 气质不统一 | 壳层组件自定义实现 |
| 路由与窗口状态双源冲突 | 刷新恢复异常 | 约定单一状态映射规则 |
| 一次迁全部模块 | 周期失控 | P0 只迁 3 个核心 app |
| 移动端硬套桌面交互 | 可用性差 | 明确单独降级策略 |

---

## 12. 冻结结论

V4 前端架构的冻结结论如下：

1. 使用独立目录 `frontend-desktop`
2. 后端 API 契约复用 V3，不在 V4 首阶段大改后端
3. 先做 Shell、Window Manager、App Registry，再迁核心模块
4. 旧 `frontend` 冻结维护，不作为桌面化重构承载体
5. 桌面端优先，移动端以降级策略收口
