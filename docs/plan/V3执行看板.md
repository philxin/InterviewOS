# InterviewOS V3 执行看板

## 1. 文档目标

用于记录 V3 当前阶段的执行状态、交付物与阻塞项，按任务流推进，不按天拆分。

V3 范围：

- 文件导入升级为 document / chunk / embedding pipeline
- 文件级知识库与最小可用检索能力
- 训练题目、提示、反馈接入 retrieval context
- 训练引用快照与结果可追溯性
- 自动知识点识别的结构预留

V3 明确不做：

- 通用文档问答平台
- 跨用户共享知识库
- 复杂 OCR
- 多 Provider 编排
- 知识图谱
- 首版复杂 rerank 链路
- 不经用户确认直接接管 `Knowledge`

---

## 2. 看板规则

状态定义：

- `TODO`：未开始
- `DOING`：进行中（最多 1-2 个并行卡片）
- `BLOCKED`：被外部依赖阻塞
- `DONE`：完成并通过当前验收标准

优先级定义：

- `P0`：阻塞 V3 主路径
- `P1`：重要但不阻塞 V3 主路径
- `P2`：增强项 / 预留项

完成标准（DoD）：

- 代码已落库或文档已冻结
- 主路径自测通过
- 错误路径有可预期返回
- 接口 / 数据 / 页面变更已同步文档
- 涉及检索的能力有用户隔离与最小可观测性

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 把“自动知识点识别”提前做到 P0 | 主路径延期，复杂度失控 | 严格以 document / chunk / retrieval 为 P0 主线 |
| 检索未按 `user_id` 过滤 | 严重数据泄露 | Repository / SQL 层强制 owner 过滤，并做双用户回归测试 |
| embedding 模型与维度过早固化错误 | 后续重建成本高 | 在 `V3-01` 冻结模型、维度、迁移策略 |
| 导入任务缺少幂等与部分失败状态 | 向量重复、用户状态误判 | 增加唯一约束、`PARTIAL` 状态与失败重试策略 |
| 评分引入动态检索后不稳定 | 用户不信任结果 | 训练时固定引用集合并持久化快照 |
| 检索结果被 prompt injection 污染 | 模型输出偏移或违规 | 引用文本进入 prompt 时明确降权为“材料”，并做基础过滤 |
