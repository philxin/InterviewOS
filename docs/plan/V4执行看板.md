# InterviewOS V4 执行看板

## 1. 文档目标

用于记录 V4 当前阶段的执行状态、交付物与阻塞项，按任务流推进，不按天拆分。

V4 范围：

- 新建 `frontend-desktop` 独立前端应用
- 建立桌面工作区壳层、Dock、Top Bar 与窗口系统
- 迁移 Knowledge / Training / History 三个核心业务 app
- 建立 Launchpad、Command Palette、Notification Center 等桌面化交互入口
- 保持旧 `frontend` 并行运行，支持灰度验证

V4 明确不做：

- 重写后端主链路
- 首版强制替换旧前端
- 100% 复刻 macOS
- 首版自由窗口缩放、多桌面 Space、复杂菜单栏系统

---

## 2. 看板规则

状态定义：

- `TODO`：未开始
- `DOING`：进行中（最多 1-2 个并行卡片）
- `BLOCKED`：被外部依赖阻塞
- `DONE`：完成并通过当前验收标准

优先级定义：

- `P0`：阻塞 V4 主路径
- `P1`：重要但不阻塞 V4 主路径
- `P2`：增强项 / 预留项

完成标准（DoD）：

- 代码已落库或文档已冻结
- 主路径自测通过
- 旧前端不受影响
- 交互边界清晰，没有把业务状态与窗口状态混写
- 涉及新增入口的文档、脚本、运行方式已同步

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 旧前端与新前端职责不清 | 维护混乱 | 明确旧版冻结维护，新版独立目录开发 |
| 壳层完成前就开始大规模迁模块 | 结构返工 | 坚持先做 Shell / Window Manager |
| 只做外观不做交互模型 | 重构价值不足 | 以 Dock、Window、Command Palette 为核心验收点 |
| 路由与窗口状态耦合失控 | 深链接与刷新恢复异常 | 先冻结状态映射规则 |
| 移动端缺少降级策略 | 上线质量不稳定 | 在 P0 阶段就定义降级边界 |
