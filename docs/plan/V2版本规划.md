# InterviewOS V2 版本规划

## 📋 版本概述

V2 不是在 V1 基础上机械叠加功能，而是将 InterviewOS 从“可演示的单人训练闭环”升级为“适合高频使用的面试训练系统”。

V2 的核心主题：

> **更快进入训练 → 更真实的面试题 → 更可信的反馈结果 → 更愿意持续训练**

V2 聚焦面试场景本身，不把产品扩展成泛知识管理平台、社交平台或复杂 AI 基础设施平台。

---

## 🎯 核心目标

### 业务目标

- [ ] 支持用户登录后持续使用，形成个人训练数据沉淀
- [ ] 降低知识录入成本，支持批量导入而不是手工逐条新增
- [ ] 让训练题更接近真实面试，而不是单一八股模板
- [ ] 让评分结果更易理解、更可信、更可操作
- [ ] 为后续高频训练闭环打下结构基础

### 产品目标

- [ ] 用户打开系统后能快速进入训练，而不是停留在配置和选择阶段
- [ ] 题目具备类型分层、难度梯度和一定的追问延展感
- [ ] 结果页先给出可理解结论，再给出改进方向
- [ ] 用户能感知自己的薄弱点与阶段进步

### 技术目标

- [ ] 在现有 Spring Security 基础上补齐用户鉴权与数据隔离
- [ ] 保持 Controller / Service / Repository 分层边界清晰
- [ ] 继续以 PostgreSQL 单库为主路径，不引入双库复杂度
- [ ] 优化 LLM 调用链路，降低题目生成与评分等待时间
- [ ] 为后续推荐训练、自动组题和回练机制预留可扩展的数据结构

## 🧭 V2 产品定位

V2 的定位不是“面试知识管理工具”，而是：

> **面向面试准备场景的高频模拟训练系统**

V2 的主线优先级：

1. 用户如何进入系统并确定训练方向
2. 系统给出的题是否足够像真实面试
3. 系统给出的反馈是否足够可信、可用

V2 的差异化方向：

1. 高频训练主链路
2. 题目真实性与训练质量
3. 评分可信度与结果反馈

## 📦 范围分层

### P0：版本骨架

P0 是 V2 必须完成的最小上线范围。

#### 1. 用户模块

- 用户注册 / 登录
- JWT 或等价 Bearer Token 鉴权
- 知识点、训练记录按用户隔离
- 未登录用户不得访问训练数据

#### 2. 训练方向初始化

- 用户首次进入后可选择目标岗位
- 岗位信息用于确定训练方向
- 暂不做复杂岗位画像与简历深度解析

#### 3. 知识录入升级

- 支持批量录入知识点
- 可支持少量高价值格式导入
- 标签体系仅保留训练筛选所需的最小结构

#### 4. 题目真实性提升

- 题目区分类型：八股题 / 项目题 / 场景题
- 题目具备难度梯度
- 尽量减少重复问法
- 允许有限追问感，但不实现完整多轮对话

#### 5. 评分反馈重构

- 分数结果转为用户可理解的档位
- 结果页先显示档位与主要问题
- 再显示缺失点与更好回答思路
- 最后给出更自然的参考表达

### P1：体验增强

P1 用于补齐影响留存的体验问题，但不应阻塞 P0 上线。

- LLM 调用性能优化
- 页面样式与训练流程优化
- 题目支持提示项，但不直接喂标准答案
- 训练结果增加最小进步感展示

### P2：差异化增强

P2 是 V2 上线后的增强方向，不作为首批强制交付范围。

- 首页优先暴露薄弱项
- 一键推荐 3-5 题训练包
- 按标签、薄弱项、历史表现自动组题
- 训练后自动更新薄弱项权重
- 回练提醒与复训机制

## 🚫 V2 明确不做

在 V2 范围内，以下内容明确不做或不深做：

| 功能 | 说明 | 处理策略 |
|------|------|----------|
| 简历深度解析 | 简历结构化抽取、项目经历识别、能力差距分析 | 不做，最多保留轻量入口 |
| 复杂岗位画像 | 岗位能力模型、岗位图谱、差距画像 | 不做 |
| 多轮对话 | 长会话追问、上下文会话管理 | 不做 |
| 复杂图表统计 | 雷达图、大屏、复杂成长曲线 | 不做 |
| 社交 / 排行榜 | 多人互动、竞争机制 | 不做 |
| 过多文件格式导入 | 大量格式适配与复杂解析链路 | 不做 |
| 超复杂标签系统 | 多层级、多维度配置型标签体系 | 不做 |
| 双数据库拆分 | MySQL + PostgreSQL 职责拆分 | 不做 |
| 复杂权限体系 | 组织级、角色级、多租户权限模型 | 不做 |
| 语义检索 / RAG | 向量检索、复杂知识召回链路 | 不做 |

## 🗂️ 关键主题

### 主题 1：用户进入门槛与训练方向

- 为什么做：V1 无用户体系，无法支持长期个人化训练
- 目标：让用户能登录、选择岗位、建立自己的训练上下文
- 风险：如果范围扩张到复杂岗位画像，会直接拖慢版本节奏

### 主题 2：题目真实性与训练质量

- 为什么做：题目太八股、太模板化会直接破坏真实感
- 目标：让题型结构、难度变化和提问风格更接近真实面试
- 风险：如果过早做成多轮对话，会显著放大实现复杂度和成本

### 主题 3：评分可信度与结果反馈

- 为什么做：评分不可信，用户就不会持续依赖系统
- 目标：把评分结果改造为“看得懂、能改进、可模仿”的反馈结构
- 风险：如果结果仍然抽象和模板化，会让 V2 核心价值失真

### 主题 4：体验基础设施

- 为什么做：题目慢、结果慢、页面粗糙都会直接影响留存
- 目标：提升训练节奏感与产品完成度
- 风险：如果忽视性能和交互优化，P0 即便做完也可能体验不达标

### 主题 5：高频训练闭环

- 为什么做：这是 InterviewOS 的长期差异化方向
- 目标：让系统主动推荐训练，而不是始终依赖用户手工选择
- 风险：如果在 V2 首批一次性做全，容易拖垮版本范围

## 🗓️ 开发阶段建议

### 阶段 1：用户与安全边界（Week 1）

- 用户实体、认证接口、Token 鉴权

- 岗位选择
- 批量知识录入

### 阶段 3：题目与评分能力重构（Week 2-3）

- 题目类型分层
- 难度梯度
- 避免重复问法
- 评分档位化与结果页结构重构

### 阶段 4：前端体验与流程优化（Week 3）

- 登录/训练入口/UI 优化
- 训练过程引导
- 结果页结构调整

### 阶段 5：性能优化与联调验收（Week 4）

- LLM 链路性能优化
- 前后端联调
- 回归测试
- 文档收口

### 阶段 6：P2 增强评估（Post V2）

- 推荐训练包
- 自动组题
- 薄弱项权重更新
- 回练提醒

## ✅ V2 完成标准

当满足以下条件时，V2 可视为完成：

### 1. 功能完整性

- 用户可以注册、登录并访问自己的训练数据
- 用户可以选择岗位方向
- 用户可以批量录入知识点
- 系统能生成不同类型、具备难度梯度的训练题
- 系统能输出新的结果页结构
- 训练数据具备用户隔离，不能跨用户访问

### 2. 技术质量

- 鉴权逻辑集中在 Security 配置 / Filter，不散落在 Controller
- Controller 不承载业务逻辑
- 错误返回保持正确 HTTP Status 语义
- 日志继续保持 requestId 串联
- 不因 V2 引入双库、多轮会话等额外复杂基础设施

### 3. 用户体验

- 用户能在较短路径内开始一次训练
- 题目不再明显重复、模板化
- 结果页用户可快速理解当前问题与改进方向
- 训练等待时间较 V1 有明显改善
- 页面无明显粗糙断层与主路径阻塞

## ⚠️ 主要风险与控制策略

### 风险 1：范围膨胀

- 表现：用户系统、岗位能力、导入、推荐、回练同时推进
- 控制：严格按 P0 / P1 / P2 分层，P2 不进入首批交付

### 风险 2：评分仍然不可信

- 表现：结果页虽然改版，但内容依然模板化
- 控制：优先重构反馈结构与 prompt 输出格式，而不是堆叠维度

### 风险 3：题目真实性不足

- 表现：题目类型有了，但问法依旧八股
- 控制：明确题型模板与生成策略，单独验收题目质量

### 风险 4：性能拖垮训练节奏

- 表现：每题生成和评分等待过长
- 控制：对 LLM 调用耗时做专项优化，必要时先做体验降级方案

### 风险 5：用户模块改造引发接口连锁变更

- 表现：前端 token、路由守卫、后端安全边界同步调整
- 控制：先冻结认证与资源边界，再推进接口联调
