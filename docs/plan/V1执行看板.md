# InterviewOS V1 执行看板

## 1. 文档目标

用于记录 V1 当前阶段的执行状态与交付物，按任务流推进，不按天拆分。

V1 范围：

- 知识点 CRUD
- AI 生成问题
- AI 评分回答
- 掌握度更新
- 训练历史查询
- 前后端闭环联调

V1 明确不做：

- 用户系统 / 权限体系
- Redis 缓存
- pgvector 语义检索
- 多轮对话
- 复杂统计与图谱

---

## 2. 看板规则

状态定义：

- `TODO`：未开始
- `DOING`：进行中（最多 1-2 个并行卡片）
- `BLOCKED`：被外部依赖阻塞
- `DONE`：完成并通过验收标准

优先级定义：

- `P0`：阻塞闭环主路径
- `P1`：重要但不阻塞主路径
- `P2`：优化项

完成标准（DoD）：

- 代码已落库
- 关键路径自测通过
- 错误路径有可预期返回
- 文档已同步（接口或行为变更时）

---

## 3. 当前基线（2026-03-07）

- 后端：仅有 Spring Boot 启动类，核心业务层未落地
- 前端：存在基础页面与路由，但与 V1 目标页面/接口不一致
- 文档：规划较完整，可直接作为实现参考

结论：当前处于 V1 开发启动期，应先完成后端骨架与核心 API，再推进前端重构与联调。

## 6. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| LLM 返回不稳定（格式漂移） | 评分解析失败、接口波动 | 评分 JSON 增加容错解析与字段兜底 |
| 文档与代码契约不一致 | 联调返工 | 以接口契约为单一事实源，变更先改文档 |
| 过早引入非 V1 范围能力 | 进度失控 | 严格执行范围清单，新增需求入 V2 |
| 前端继续沿用旧类型模型 | 大量无效 API 调用 | 先完成 `V1-07` 再开发页面 |
