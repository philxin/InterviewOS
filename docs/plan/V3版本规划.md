# InterviewOS V3 版本规划

## 📋 版本概述

V3 不是简单给 V2 加一个“向量检索”按钮，而是把 InterviewOS 从“基于单条知识点训练”升级为“基于文件级知识库做检索增强训练”。

V3 的核心主题：

> **用户导入资料后，系统自动建立可检索的知识底座，并将这些材料用于题目生成、提示与答案校准**

V3 依然服务于面试训练场景本身，不把产品扩展成通用知识库平台、文档协作平台或复杂 Agent 编排平台。

---

## 🎯 核心目标

### 业务目标

- [ ] 降低用户整理资料的成本，从“逐条录入知识点”升级为“导入文件后自动建立训练上下文”
- [ ] 提升题目与反馈的可信度，让系统输出不再只依赖单条 `Knowledge.content`
- [ ] 为后续知识点抽取、训练推荐、答案校准建立统一数据底座
- [ ] 保持用户级数据隔离，确保导入资料不会被跨用户召回

### 产品目标

- [ ] 用户上传文件后能看到明确的处理进度，而不是只有成功 / 失败二元结果
- [ ] 用户在训练时能感知“本题 / 本次反馈依据了哪些材料”
- [ ] 当检索不到足够证据时，系统应明确降级，而不是伪造依据
- [ ] 用户可以逐步从“资料导入”过渡到“自动发现知识点”，而不是一次性全自动接管

### 技术目标

- [ ] 在现有 `KnowledgeFileImport` 基础上升级为可重试、可观测的导入 pipeline
- [ ] 在 PostgreSQL + pgvector 上落地文档切片与向量索引
- [ ] 在现有 Spring AI 能力上补齐 embedding 与 retrieval 服务
- [ ] 保持 Controller / Service / Repository 分层边界清晰
- [ ] 为模型版本切换、向量重建、召回回放测试预留结构基础

---

## 🧭 V3 产品定位

V3 的定位不是“知识管理平台”，而是：

> **面向面试训练场景的文件级知识库与检索增强训练系统**

V3 的主线优先级：

1. 用户上传的资料能否稳定转化为可检索的知识底座
2. 训练题目、提示和反馈能否基于真实材料做校准
3. 系统是否能解释“为什么这样出题 / 为什么这样给反馈”

V3 的差异化方向：

1. 文件导入后的自动结构化处理
2. 训练链路中的可追溯引用
3. 面向训练场景的 RAG，而不是泛问答型 RAG

---

## 📦 范围分层

### P0：V3 最小上线范围

P0 的目标是先把“文件级知识库 + 检索增强训练”主链路跑通。

#### 1. 文件导入链路升级

- 上传文件后，异步执行解析、清洗、切片、向量化
- 导入任务支持阶段化状态，而不是只有 `PENDING / PROCESSING / SUCCESS / FAILED`
- 返回 chunk 总数、已向量化数量、失败数量等进度信息
- 同一个导入任务必须具备幂等与安全重试能力

#### 2. 文档与切片索引

- 新增文档主表，承接文件级元信息
- 新增 chunk 表，存储切片文本、位置信息和 embedding
- 向量记录必须带 `user_id`、模型名、维度、状态信息
- 允许后续对文档重切片、重向量化，而不破坏训练主表

#### 3. 检索能力落地

- 提供最小可用的检索 API
- 支持按当前用户范围检索
- 返回引用片段、来源文档、页码 / 段落等信息
- 检索无命中或低命中时支持显式降级

#### 4. 训练主链路接入 RAG

- 题目生成支持基于检索片段做上下文增强
- 提示生成支持基于检索片段做方向校准
- 答案反馈支持基于固定引用集合做校准
- 训练记录需要能追溯本题使用了哪些引用片段

#### 5. 安全与可观测性

- 向量检索必须按 `user_id` 做硬过滤
- 文档内容进入 prompt 时必须以“引用材料”而非“可执行指令”处理
- 记录检索 topK、距离分布、引用 chunk 列表、是否触发降级

### P1：自动知识点识别

P1 在 P0 的文档与 chunk 基础上增加“自动识别知识点”能力，但不直接替代现有 `Knowledge` 主模型。

- 从文档或 chunk 中抽取概念 / 知识点候选
- 生成候选标题、摘要、别名、支持片段
- 允许用户确认后转为正式 `Knowledge`
- 允许将候选知识点与现有知识点做合并或忽略

### P2：质量与工程化增强

- embedding 缓存与批量调用优化
- chunk 去重、低质量内容过滤、相似度聚类
- 向量重建与模型版本切换机制
- 检索质量回放测试与离线评测
- 结果页展示更丰富的引用解释与调试信息

## 🚫 V3 明确不做

在 V3 范围内，以下内容明确不做或不深做：

| 功能 | 说明 | 处理策略 |
|------|------|----------|
| 通用 AI 知识库平台 | 脱离面试训练场景的泛文档问答 | 不做 |
| 外部网页抓取 | 自动抓取网页、博客、仓库做知识源 | 不做 |
| 复杂 OCR | 扫描件、图片型 PDF 的重度 OCR 适配 | 不做，后续评估 |
| 多 Provider 编排 | 多 embedding / 多 reranker / 多模型动态路由 | 不做 |
| 共享知识库 | 跨用户共享资料、团队级知识空间 | 不做 |
| 知识图谱 | 自动实体关系图、图谱推理 | 不做 |
| 完整自动化知识点接管 | 不经用户确认直接覆盖现有 `Knowledge` | 不做 |
| 复杂 rerank 链路 | 召回 + rerank + 多阶段融合检索 | 首版不做 |
| 全量历史数据回填 | 一次性把所有旧 `Knowledge` 反向索引化 | 不做，按需补做 |

## 🗂️ 关键主题

### 主题 1：文件级知识库

- 为什么做：V2 的文件导入最终还是落成一条完整 `Knowledge`，无法支撑细粒度检索
- 目标：把文件转为文档与 chunk，而不是直接塞回整段文本
- 风险：如果不拆文档层与 chunk 层，后续检索、重建、追溯都会变得混乱

### 主题 2：检索增强训练

- 为什么做：当前训练题目和反馈仍然只依赖 `Knowledge` 主体文本
- 目标：让问题、提示、反馈都能基于真实检索证据做校准
- 风险：如果每次动态召回不稳定，会导致同题不同评分、用户不信任

### 主题 3：安全边界

- 为什么做：RAG 一旦跨用户召回，就是严重数据泄露
- 目标：所有检索都在用户边界内执行，并防范 prompt injection
- 风险：如果只在 Controller 层做过滤，最终很容易漏到 Repository / SQL 层

### 主题 4：导入可观测性与幂等

- 为什么做：文件导入会比 V2 的单步建 Knowledge 复杂得多
- 目标：支持进度查询、失败定位、部分成功和单步重试
- 风险：如果没有幂等和阶段状态，重试就会制造重复向量和脏数据

### 主题 5：自动知识点识别

- 为什么做：这是 V3 后续最有价值的体验升级方向
- 目标：从导入材料中自动发现候选知识点，降低用户整理成本
- 风险：如果在 P0 就强行自动接管 `Knowledge`，会破坏现有训练主链路的稳定性

## 🗓️ 开发阶段建议

### 阶段 1：文档冻结与数据模型设计（Week 1）

- 冻结 V3 边界与不做项
- 冻结文档、chunk、引用快照的表结构
- 冻结切片策略、embedding 模型配置项、低召回降级规则

### 阶段 2：导入 pipeline 升级（Week 1-2）

- 解析文本清洗
- 文档落库
- chunk 切分
- embedding 批处理
- 导入状态与进度查询

### 阶段 3：检索与训练链路接入（Week 2-3）

- 新增 retrieval service
- 新增检索 API
- 训练题目生成接入 retrieval context
- 反馈生成接入固定引用快照

### 阶段 4：前端接入与联调（Week 3-4）

- 文件导入进度页升级
- 文档列表与最小检索验证页
- 训练题目与结果页展示引用依据

### 阶段 5：安全、回归与质量验证（Week 4）

- 双用户隔离回归
- 导入重试 / 部分失败验证
- 低召回降级验证
- 检索 trace 指标校验

### 阶段 6：P1 评估（Post V3 P0）

- 自动知识点识别
- 候选知识点确认流
- chunk 去重与召回质量优化

## ✅ V3 P0 完成标准

当满足以下条件时，V3 P0 可视为完成：

### 1. 文件处理完整性

- 用户可上传支持格式文件并查看阶段化进度
- 文档可被切成稳定 chunk 并成功写入向量索引
- 导入失败时能返回可解释原因

### 2. 检索可用性

- 当前用户可基于自然语言查询召回自己的文档片段
- 检索结果包含足够的来源信息用于展示与追溯
- 检索低命中时系统能明确降级

### 3. 训练链路增强

- 题目生成、提示、反馈至少有一条主链路接入 retrieval context
- 训练记录可回看引用过哪些片段
- 同一题的反馈依据可复现，不因再次检索漂移

### 4. 安全与稳定性

- 双用户数据绝不交叉召回
- 导入重试不会产生重复 chunk / duplicate embedding
- 日志不打印原文正文，只记录 fingerprint、长度、耗时、引用 id

### 5. 可观测性

- 至少可观察到空召回比例、平均召回数、平均导入耗时
- 至少可追踪每次训练调用使用了哪些 chunk
