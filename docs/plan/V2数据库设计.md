# InterviewOS V2 数据库设计

## 📊 概述

V2 数据库设计目标不是单纯在 V1 的两张表上打补丁，而是为以下能力提供稳定数据基础：

1. 用户鉴权与数据隔离
2. 知识点批量导入与标签组织
3. 会话化训练流程
4. 更可解释的评分反馈结构
5. 后续推荐训练与回练机制的扩展空间

V2 继续以 **PostgreSQL 单库** 为主落地路径，不引入 MySQL + PostgreSQL 双库拆分。

---

## 🧭 设计原则

### 1. 用户隔离优先

- 所有业务数据必须归属于具体用户
- 查询与写入默认带 `user_id` 维度
- 禁止通过纯资源主键直接暴露跨用户数据

### 2. 为当前版本服务，不做过度设计

- 不为 V2 引入复杂岗位画像
- 不为 V2 引入 RAG / 向量检索表
- 不为 V2 引入复杂权限模型

### 3. 为后续闭环预留扩展点

- 训练流程改为 `session + question`
- 评分结构支持档位化和可解释反馈
- 首页薄弱项与推荐训练未来可基于现有字段扩展

### 4. 尽量减少破坏性耦合

- 保留 `knowledge.mastery` 作为当前主状态
- 训练历史细节沉淀在训练题目明细表
- 标签单独建表，不把复杂数组逻辑塞进核心主表

---

## 🔄 V2 相比 V1 的核心变化

### V1

- `knowledge`
- `training_record`

### V2

- `app_user`
- `knowledge`
- `knowledge_tag`
- `training_session`
- `training_question`
- `knowledge_file_import`（P1）

### 变化说明

1. `knowledge` 从公共数据变为用户私有数据
2. `training_record` 不再直接承载整个训练过程，改由 `training_session + training_question` 表达
3. 评分结果不再只保存三维分数，还需要保存档位、主要问题、缺失点和回答思路
4. 标签体系采用独立表，支持后续筛选与推荐

## 🧱 表结构总览

| 表名 | 用途 | 优先级 |
|------|------|--------|
| `app_user` | 用户账号与训练方向 | P0 |
| `knowledge` | 用户知识点主表 | P0 |
| `knowledge_tag` | 知识点标签表 | P0 |
| `training_session` | 一次训练会话 | P0 |
| `training_question` | 会话下的问题、回答与反馈 | P0 |
| `knowledge_file_import` | 文件导入任务记录 | P1 |

## 1. `app_user`

存储用户账户与训练方向信息。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `bigserial` | PK | 主键 |
| `email` | `varchar(255)` | NOT NULL, UNIQUE | 登录邮箱 |
| `password_hash` | `varchar(255)` | NOT NULL | 密码哈希 |
| `display_name` | `varchar(50)` | NOT NULL | 展示名称 |
| `target_role` | `varchar(50)` | NULL | 训练方向，限定为 `TargetRole` 枚举 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |
| `updated_at` | `timestamp` | NOT NULL | 更新时间 |

**索引**

- `uk_app_user_email`

**业务规则**

- 邮箱全局唯一
- 密码只存哈希，不存明文
- `target_role` 只承载训练方向，不承载复杂岗位画像

---

## 2. `knowledge`

存储当前用户的知识点主数据。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `bigserial` | PK | 主键 |
| `user_id` | `bigint` | NOT NULL, FK | 所属用户 |
| `title` | `varchar(200)` | NOT NULL | 标题 |
| `content` | `text` | NOT NULL | 内容 |
| `mastery` | `integer` | NOT NULL, DEFAULT 0 | 掌握度，0-100 |
| `source_type` | `varchar(30)` | NOT NULL | 来源类型 |
| `status` | `varchar(20)` | NOT NULL, DEFAULT `ACTIVE` | 知识点状态 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |
| `updated_at` | `timestamp` | NOT NULL | 更新时间 |
| `archived_at` | `timestamp` | NULL | 归档时间 |

**外键**

- `fk_knowledge_user_id`: `user_id -> app_user(id)`

**索引**

- `idx_knowledge_user_created`：`(user_id, created_at desc)`
- `idx_knowledge_user_mastery`：`(user_id, mastery asc, updated_at desc)`
- `idx_knowledge_user_status_created`：`(user_id, status, created_at desc)`

**业务规则**

- `mastery` 范围固定为 `0-100`
- 默认创建时 `mastery = 0`
- 同一用户下知识点允许重名，但不建议前端默认鼓励
- `status` 建议值：
  - `ACTIVE`
  - `ARCHIVED`
- `source_type` 建议值：
  - `MANUAL`
  - `BATCH_IMPORT`
  - `FILE_IMPORT`
  - `ROLE_GENERATED`
- V2 中知识点删除语义为“归档”，不做物理删除

---

## 3. `knowledge_tag`

存储知识点标签，用于筛选、批量组织和后续推荐。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `bigserial` | PK | 主键 |
| `knowledge_id` | `bigint` | NOT NULL, FK | 关联知识点 |
| `tag` | `varchar(50)` | NOT NULL | 标签值 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |

**外键**

- `fk_knowledge_tag_knowledge_id`: `knowledge_id -> knowledge(id)`

**索引**

- `uk_knowledge_tag_unique`：`(knowledge_id, tag)` 唯一
- `idx_knowledge_tag_tag`：`(tag)`

**业务规则**

- 标签入库前统一 `trim + lowercase`
- V2 只支持简单平铺标签，不做多层级标签树
- 获取当前用户标签列表时，通过 `knowledge -> knowledge_tag` 聚合去重

---

## 4. `training_session`

存储一次训练会话的元信息。  
V2 虽然主路径仍以单次题目训练为主，但表结构按“一个 session 下可有多题”设计，为 P2 推荐训练包预留扩展。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `uuid` | PK | 会话主键 |
| `user_id` | `bigint` | NOT NULL, FK | 所属用户 |
| `knowledge_id` | `bigint` | NOT NULL, FK | 本次训练主知识点 |
| `question_type` | `varchar(20)` | NOT NULL | 题型 |
| `difficulty` | `varchar(20)` | NOT NULL | 难度 |
| `hint_enabled` | `boolean` | NOT NULL, DEFAULT true | 是否允许提示 |
| `status` | `varchar(20)` | NOT NULL | `IN_PROGRESS / COMPLETED / ABANDONED` |
| `total_questions` | `integer` | NOT NULL, DEFAULT 1 | 总题数 |
| `answered_questions` | `integer` | NOT NULL, DEFAULT 0 | 已回答题数 |
| `current_question_no` | `integer` | NOT NULL, DEFAULT 1 | 当前题序 |
| `summary_score` | `integer` | NULL | session 聚合分 |
| `summary_band` | `varchar(20)` | NULL | session 聚合档位 |
| `summary_major_issue` | `varchar(255)` | NULL | session 聚合主要问题 |
| `started_at` | `timestamp` | NOT NULL | 开始时间 |
| `completed_at` | `timestamp` | NULL | 完成时间 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |

**外键**

- `fk_training_session_user_id`: `user_id -> app_user(id)`
- `fk_training_session_knowledge_id`: `knowledge_id -> knowledge(id)`

**索引**

- `idx_training_session_user_created`：`(user_id, created_at desc)`
- `idx_training_session_user_status`：`(user_id, status, created_at desc)`
- `idx_training_session_knowledge_created`：`(knowledge_id, created_at desc)`

**业务规则**

- `knowledge_id` 必须属于 `user_id`
- `status = COMPLETED` 时必须存在 `completed_at`
- `total_questions >= 1`
- `answered_questions >= 0`
- V2 P0 默认 `total_questions = 1`

---

## 5. `training_question`

存储某次训练会话中的具体问题、回答和反馈结果。  
这是 V2 的核心事实表。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `uuid` | PK | 问题主键 |
| `session_id` | `uuid` | NOT NULL, FK | 所属训练会话 |
| `knowledge_id` | `bigint` | NOT NULL, FK | 冗余保存对应知识点，便于查询 |
| `order_no` | `integer` | NOT NULL | 会话内题序，从 1 开始 |
| `parent_question_id` | `uuid` | NULL, FK | 追问来源问题 |
| `question_type` | `varchar(20)` | NOT NULL | 题型 |
| `difficulty` | `varchar(20)` | NOT NULL | 难度 |
| `question_text` | `text` | NOT NULL | 问题正文 |
| `hint_text` | `text` | NULL | 最近一次提示内容 |
| `hint_used` | `boolean` | NOT NULL, DEFAULT false | 是否使用过提示 |
| `answer_text` | `text` | NULL | 用户回答 |
| `score` | `integer` | NULL | 最终分数，0-100 |
| `feedback_band` | `varchar(20)` | NULL | 档位 |
| `major_issue` | `varchar(255)` | NULL | 主要问题摘要 |
| `missing_points` | `jsonb` | NULL | 缺失点数组 |
| `better_answer_approach` | `jsonb` | NULL | 更好回答思路数组 |
| `natural_example_answer` | `text` | NULL | 更自然的参考表达 |
| `weak_tags` | `jsonb` | NULL | 暴露出的薄弱标签数组 |
| `mastery_before` | `integer` | NULL | 本次训练前掌握度 |
| `mastery_after` | `integer` | NULL | 本次训练后掌握度 |
| `answered_at` | `timestamp` | NULL | 回答提交时间 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |

**外键**

- `fk_training_question_session_id`: `session_id -> training_session(id)`
- `fk_training_question_knowledge_id`: `knowledge_id -> knowledge(id)`
- `fk_training_question_parent_id`: `parent_question_id -> training_question(id)`

**索引**

- `uk_training_question_session_order`：`(session_id, order_no)` 唯一
- `idx_training_question_knowledge_created`：`(knowledge_id, created_at desc)`
- `idx_training_question_band_created`：`(feedback_band, created_at desc)`

**业务规则**

- `order_no >= 1`
- `score` 范围固定为 `0-100`
- `mastery_before / mastery_after` 范围固定为 `0-100`
- `parent_question_id` 仅用于轻量追问链，不代表完整多轮会话系统
- `missing_points / better_answer_approach / weak_tags` 统一存 JSON 数组
- `feedback_band` 必须与服务端 `score -> band` 映射规则保持一致

**为什么使用 JSONB**

- 这些字段天然是小数组
- 使用 JSONB 比额外拆 3-4 张子表更轻量
- 当前主路径是 PostgreSQL，符合 V2 设计前提

## 6. `knowledge_file_import`（P1）

存储文件导入任务状态，用于异步导入场景。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| `id` | `uuid` | PK | 导入任务主键 |
| `user_id` | `bigint` | NOT NULL, FK | 所属用户 |
| `file_name` | `varchar(255)` | NOT NULL | 原始文件名 |
| `content_type` | `varchar(100)` | NOT NULL | MIME 类型 |
| `file_size` | `bigint` | NOT NULL | 文件大小 |
| `status` | `varchar(20)` | NOT NULL | `PENDING / PROCESSING / SUCCESS / FAILED` |
| `default_tags` | `jsonb` | NULL | 默认标签数组 |
| `created_count` | `integer` | NOT NULL, DEFAULT 0 | 导入成功数量 |
| `failure_reason` | `text` | NULL | 失败原因摘要 |
| `created_at` | `timestamp` | NOT NULL | 创建时间 |
| `updated_at` | `timestamp` | NOT NULL | 更新时间 |
| `completed_at` | `timestamp` | NULL | 完成时间 |

**外键**

- `fk_knowledge_file_import_user_id`: `user_id -> app_user(id)`

**索引**

- `idx_file_import_user_created`：`(user_id, created_at desc)`
- `idx_file_import_status_created`：`(status, created_at desc)`

**业务规则**

- V2 只支持少量高价值格式
- 导入失败时应记录可读摘要，不记录敏感文件正文

## 📝 推荐 SQL 草案

```sql
CREATE TABLE app_user (
    id BIGSERIAL PRIMARY KEY,
    email VARCHAR(255) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    display_name VARCHAR(50) NOT NULL,
    target_role VARCHAR(50),
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT chk_app_user_target_role CHECK (
        target_role IS NULL OR target_role IN (
            'JAVA_BACKEND', 'FRONTEND', 'FULLSTACK', 'DEVOPS', 'DATA_ENGINEER'
        )
    )
);

CREATE TABLE knowledge (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    mastery INTEGER NOT NULL DEFAULT 0,
    source_type VARCHAR(30) NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    archived_at TIMESTAMP NULL,
    CONSTRAINT fk_knowledge_user_id
        FOREIGN KEY (user_id) REFERENCES app_user(id) ON DELETE CASCADE,
    CONSTRAINT chk_knowledge_mastery_range CHECK (mastery >= 0 AND mastery <= 100),
    CONSTRAINT chk_knowledge_source_type CHECK (
        source_type IN ('MANUAL', 'BATCH_IMPORT', 'FILE_IMPORT', 'ROLE_GENERATED')
    ),
    CONSTRAINT chk_knowledge_status CHECK (
        status IN ('ACTIVE', 'ARCHIVED')
    )
);

CREATE TABLE knowledge_tag (
    id BIGSERIAL PRIMARY KEY,
    knowledge_id BIGINT NOT NULL,
    tag VARCHAR(50) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT fk_knowledge_tag_knowledge_id
        FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    CONSTRAINT uk_knowledge_tag_unique UNIQUE (knowledge_id, tag)
);

CREATE TABLE training_session (
    id UUID PRIMARY KEY,
    user_id BIGINT NOT NULL,
    knowledge_id BIGINT NOT NULL,
    question_type VARCHAR(20) NOT NULL,
    difficulty VARCHAR(20) NOT NULL,
    hint_enabled BOOLEAN NOT NULL DEFAULT TRUE,
    status VARCHAR(20) NOT NULL,
    total_questions INTEGER NOT NULL DEFAULT 1,
    answered_questions INTEGER NOT NULL DEFAULT 0,
    current_question_no INTEGER NOT NULL DEFAULT 1,
    summary_score INTEGER NULL,
    summary_band VARCHAR(20) NULL,
    summary_major_issue VARCHAR(255) NULL,
    started_at TIMESTAMP NOT NULL DEFAULT NOW(),
    completed_at TIMESTAMP NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT fk_training_session_user_id
        FOREIGN KEY (user_id) REFERENCES app_user(id) ON DELETE CASCADE,
    CONSTRAINT fk_training_session_knowledge_id
        FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    CONSTRAINT chk_training_session_total_questions CHECK (total_questions >= 1),
    CONSTRAINT chk_training_session_answered_questions CHECK (answered_questions >= 0),
    CONSTRAINT chk_training_session_current_question_no CHECK (current_question_no >= 1),
    CONSTRAINT chk_training_session_status CHECK (
        status IN ('IN_PROGRESS', 'COMPLETED', 'ABANDONED')
    ),
    CONSTRAINT chk_training_session_question_type CHECK (
        question_type IN ('FUNDAMENTAL', 'PROJECT', 'SCENARIO')
    ),
    CONSTRAINT chk_training_session_difficulty CHECK (
        difficulty IN ('EASY', 'MEDIUM', 'HARD')
    ),
    CONSTRAINT chk_training_session_summary_score CHECK (
        summary_score IS NULL OR (summary_score >= 0 AND summary_score <= 100)
    ),
    CONSTRAINT chk_training_session_summary_band CHECK (
        summary_band IS NULL OR summary_band IN ('UNCLEAR', 'INCOMPLETE', 'BASIC', 'GOOD', 'STRONG')
    )
);

CREATE TABLE training_question (
    id UUID PRIMARY KEY,
    session_id UUID NOT NULL,
    knowledge_id BIGINT NOT NULL,
    order_no INTEGER NOT NULL,
    parent_question_id UUID NULL,
    question_type VARCHAR(20) NOT NULL,
    difficulty VARCHAR(20) NOT NULL,
    question_text TEXT NOT NULL,
    hint_text TEXT NULL,
    hint_used BOOLEAN NOT NULL DEFAULT FALSE,
    answer_text TEXT NULL,
    score INTEGER NULL,
    feedback_band VARCHAR(20) NULL,
    major_issue VARCHAR(255) NULL,
    missing_points JSONB NULL,
    better_answer_approach JSONB NULL,
    natural_example_answer TEXT NULL,
    weak_tags JSONB NULL,
    mastery_before INTEGER NULL,
    mastery_after INTEGER NULL,
    answered_at TIMESTAMP NULL,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT fk_training_question_session_id
        FOREIGN KEY (session_id) REFERENCES training_session(id) ON DELETE CASCADE,
    CONSTRAINT fk_training_question_knowledge_id
        FOREIGN KEY (knowledge_id) REFERENCES knowledge(id) ON DELETE CASCADE,
    CONSTRAINT fk_training_question_parent_id
        FOREIGN KEY (parent_question_id) REFERENCES training_question(id) ON DELETE SET NULL,
    CONSTRAINT uk_training_question_session_order UNIQUE (session_id, order_no),
    CONSTRAINT chk_training_question_order_no CHECK (order_no >= 1),
    CONSTRAINT chk_training_question_question_type CHECK (
        question_type IN ('FUNDAMENTAL', 'PROJECT', 'SCENARIO')
    ),
    CONSTRAINT chk_training_question_difficulty CHECK (
        difficulty IN ('EASY', 'MEDIUM', 'HARD')
    ),
    CONSTRAINT chk_training_question_score_range CHECK (score IS NULL OR (score >= 0 AND score <= 100)),
    CONSTRAINT chk_training_question_feedback_band CHECK (
        feedback_band IS NULL OR feedback_band IN ('UNCLEAR', 'INCOMPLETE', 'BASIC', 'GOOD', 'STRONG')
    ),
    CONSTRAINT chk_training_question_mastery_before_range CHECK (
        mastery_before IS NULL OR (mastery_before >= 0 AND mastery_before <= 100)
    ),
    CONSTRAINT chk_training_question_mastery_after_range CHECK (
        mastery_after IS NULL OR (mastery_after >= 0 AND mastery_after <= 100)
    )
);
```

## 🔍 常用查询示例

### 1. 查询当前用户知识点列表

```sql
SELECT *
FROM knowledge
WHERE user_id = :userId
  AND status = 'ACTIVE'
ORDER BY created_at DESC;
```

### 2. 查询当前用户薄弱知识点

```sql
SELECT *
FROM knowledge
WHERE user_id = :userId
  AND status = 'ACTIVE'
ORDER BY mastery ASC, updated_at DESC
LIMIT 10;
```

### 3. 查询某知识点训练历史

```sql
SELECT tq.*, ts.status
FROM training_question tq
JOIN training_session ts ON ts.id = tq.session_id
WHERE ts.user_id = :userId
  AND tq.knowledge_id = :knowledgeId
ORDER BY tq.created_at DESC;
```

### 4. 聚合当前用户所有标签

```sql
SELECT DISTINCT kt.tag
FROM knowledge_tag kt
JOIN knowledge k ON k.id = kt.knowledge_id
WHERE k.user_id = :userId
ORDER BY kt.tag ASC;
```

### 5. 查询最近训练结果

```sql
SELECT tq.id, tq.score, tq.feedback_band, tq.major_issue, tq.created_at
FROM training_question tq
JOIN training_session ts ON ts.id = tq.session_id
WHERE ts.user_id = :userId
ORDER BY tq.created_at DESC
LIMIT 20;
```

## 🔁 迁移建议

### 1. V2 首次迁移顺序

1. 创建 `app_user`
2. 改造 `knowledge`，增加 `user_id / source_type / updated_at`
3. 创建 `knowledge_tag`
4. 创建 `training_session`
5. 创建 `training_question`
6. 创建 `knowledge_file_import`（如 P1 同步推进）

### 2. V1 数据处理策略

由于 V1 没有用户体系，历史 `knowledge / training_record` 数据无法天然推断 owner。推荐：

- **开发环境**：允许重建本地数据
- **测试环境**：可统一挂到一个 bootstrap 用户下
- **生产环境**：若未来存在真实 V1 用户数据，必须先定义迁移归属规则，再执行迁移

### 3. `training_record` 处理建议

- 不建议继续在 V2 复用 `training_record`
- 推荐在迁移窗口内保留旧表只读
- 待新接口与新实体稳定后，再清理旧表

### 4. 时间字段维护建议

- `updated_at` 推荐通过应用层 `@PreUpdate` 维护
- 如后续引入 SQL migration 工具，可补充数据库触发器统一更新时间字段

---

## ⚠️ 风险与约束

### 风险 1：标签设计过重

- 控制策略：V2 仅使用平铺标签表，不做树形结构

### 风险 2：评分字段设计失控

- 控制策略：只保留结果页真正需要的结构化字段，不把全部 LLM 原始输出落库

### 风险 3：会话设计过早追求复杂多题模式

- 控制策略：P0 默认每 session 一题，但表结构允许扩展

### 风险 4：旧数据迁移复杂

- 控制策略：优先保证 V2 新链路正确，旧数据迁移按环境分别处理
