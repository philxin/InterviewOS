# InterviewOS V4 版本规划

## 1. 版本概述

V4 的核心主题不是继续给当前前端追加页面，而是新建一套独立的 `frontend-desktop`，把 InterviewOS 从“页面式 Web 应用”升级为“桌面化工作区”。

V4 的核心主题：

> **以 macOS desktop workspace 为灵感，重写前端壳层与交互模型，让知识管理、训练、复盘和邀请管理都运行在统一的桌面工作区中**

V4 只重构前端表现层与信息架构，不主动重写 V3 已稳定的后端契约。

---

## 2. 核心目标

### 2.1 业务目标

- [ ] 在不破坏 V3 主链路的前提下，为 InterviewOS 提供更强的长期使用体验
- [ ] 把知识点、文档库、训练、历史、邀请等模块收敛为统一工作区，而不是分散页面
- [ ] 降低用户在多模块之间切换的成本，提升训练效率与沉浸感
- [ ] 为后续增加更多 AI 工作流预留更强的前端壳层能力

### 2.2 产品目标

- [ ] 登录后进入桌面而不是传统 dashboard 页面
- [ ] 支持 Dock、Launchpad、窗口切换和工作区语义
- [ ] 让核心任务支持多窗口并行，而不是单页面来回跳转
- [ ] 引入全局命令入口，减少深层导航依赖
- [ ] 桌面端获得完整体验，移动端获得明确降级版本

### 2.3 技术目标

- [ ] 新建 `frontend-desktop` 独立应用目录，不污染现有 `frontend`
- [ ] 复用 V3 API 契约与 token 机制，避免后端重构耦合
- [ ] 建立桌面壳层、窗口管理、应用注册表、全局状态管理等基础设施
- [ ] 控制重构范围，先交付桌面壳层和 3 个核心 app，再逐步迁移其他模块
- [ ] 建立可灰度切换的部署策略，支持旧前端与新前端并行

---

## 3. V4 产品定位

V4 的定位不是“做一个像 macOS 的皮肤”，而是：

> **做一套适合 InterviewOS 的桌面工作区，让复杂训练任务以系统级工作流方式被组织起来**

V4 的主线优先级：

1. 先建立桌面工作区壳层
2. 再建立窗口与应用模型
3. 再迁移核心业务 app
4. 最后补命令面板、通知中心、个性化与移动端降级

V4 的差异化方向：

1. 面向训练场景的 workspace，而不是普通后台系统
2. 把知识、文档、训练、历史统一到多任务工作流中
3. 以桌面端沉浸体验驱动产品感知升级

---

## 4. 范围分层

### P0：V4 最小上线范围

P0 的目标是先把 `frontend-desktop` 的桌面壳层与 3 个核心业务 app 跑通。

#### 4.1 新前端应用骨架

- 新建 `frontend-desktop/`
- 建立 `Vue 3 + TypeScript + Vite + Pinia + Vue Router + Axios` 基础工程
- 接入与现有后端一致的 `Authorization: Bearer <token>` 机制
- 独立维护环境变量、构建脚本与 README

#### 4.2 Desktop Shell

- 登录后进入桌面主页
- 提供桌面壁纸、Top Bar、Dock、桌面图标区
- 建立统一主题变量与基础动效系统
- 保留 InterviewOS 自身品牌表达，不做 Apple 1:1 复刻

#### 4.3 Window Manager

- 每个业务模块按“应用窗口”打开，而不是整页跳转
- 支持打开、关闭、聚焦、最小化、最大化
- 支持窗口层级管理与最近使用切换
- P0 不做自由缩放，只做固定尺寸或有限尺寸方案

#### 4.4 App Registry 与核心模块迁移

P0 首批迁移：

- `Knowledge Explorer`
- `Training Console`
- `History Review`

这些模块在新壳层中必须能独立打开，并跑通主链路 API。

#### 4.5 路由与深链接策略

- 支持最小可用的深链接
- 保持分享链接和刷新恢复的基本能力
- 路由作为“窗口状态映射”，不再等价于完整页面结构

### P1：系统感增强

- Launchpad
- Command Palette (`Cmd/Ctrl + K`)
- Notification Center
- Recent Apps / Recent Files
- 窗口位置与大小记忆
- Dashboard widgets 桌面化

### P2：完善与扩展

- Settings 应用与主题切换
- 移动端降级壳层
- 快捷键体系
- 动效细节优化
- 性能优化与埋点
- 邀请管理、文档库、候选知识点等剩余模块迁移

## 5. V4 明确不做

| 功能 | 说明 | 处理策略 |
|------|------|----------|
| 重写后端主链路 | V3 后端契约已稳定 | 不做 |
| 强制替换旧前端 | 首阶段保留 `frontend` 并行运行 | 不做 |
| 100% 复刻 macOS | 视觉和行为可借鉴，但不做像素级仿制 | 不做 |
| 首版自由窗口缩放 | 成本高、收益一般 | 不做 |
| 首版复杂菜单栏系统 | Apple 风格全局菜单维护成本高 | 不做 |
| 首版多桌面 Space 系统 | 复杂度过高 | 不做 |
| 首版离线模式 | 当前价值不足 | 不做 |

## 7. 关键主题

### 主题 1：独立前端重写

- 为什么做：现有 `frontend` 已承载完整业务，但不适合继续承接桌面交互模型
- 目标：新建 `frontend-desktop`，让重构与生产版本解耦
- 风险：双前端并存期间需要管理文档、构建和部署入口

### 主题 2：桌面工作区壳层

- 为什么做：桌面化体验的本质是壳层与交互模型，不是视觉皮肤
- 目标：建立 Top Bar、Dock、桌面图标和窗口系统
- 风险：如果只是把旧页面塞进毛玻璃容器，会变成高成本换皮

### 主题 3：窗口与应用模型

- 为什么做：业务模块已经超过简单导航条的舒适范围
- 目标：把知识、训练、历史等模块建模为“应用”而不是“页面”
- 风险：如果窗口状态与路由状态没有统一约束，刷新恢复和分享链接会变脆弱

### 主题 4：新旧前端并行与灰度

- 为什么做：V3 主链路已经可用，不能让前端重构阻塞现有业务
- 目标：保留旧 `frontend`，为新前端提供灰度入口
- 风险：如果强切入口，任何桌面壳层 bug 都会直接影响主用户链路

### 主题 5：桌面优先与移动端降级

- 为什么做：桌面工作区范式天然更适合 PC，不适合原样搬到手机
- 目标：桌面端交付完整体验，移动端交付清晰降级策略
- 风险：如果一套交互硬兼容所有屏幕，会两边都不好用

## 8. 开发阶段建议

### 阶段 1：文档冻结与设计决策（Week 1）

- 冻结 V4 边界与不做项
- 冻结 `frontend-desktop` 目录与技术选型
- 冻结视觉方向、窗口能力边界、首批迁移模块
- 冻结新旧前端并行与灰度策略

### 阶段 2：基础工程与 Shell 骨架（Week 1-2）

- 初始化 Vite 项目
- 建立设计 token、Desktop Shell、Top Bar、Dock
- 建立应用注册表与 window store
- 打通登录态与 API client

### 阶段 3：窗口系统与核心 app 迁移（Week 2-4）

- 实现 Window Manager
- 迁移 Knowledge Explorer
- 迁移 Training Console
- 迁移 History Review

### 阶段 4：系统增强（Week 4-5）

- Launchpad
- Command Palette
- Notification Center
- 基础快捷键与窗口记忆

### 阶段 5：灰度、回归与移动端降级（Week 5-6）

- 双前端并行入口验证
- 桌面端体验回归
- 移动端降级方案落地
- 文档与部署说明收口

## 10. V4 P0 完成标准

当满足以下条件时，V4 P0 可视为完成：

### 10.1 工程完整性

- `frontend-desktop` 可独立启动与构建
- Token 与 API baseURL 配置可复用现有后端契约
- 新旧前端可并行存在，不互相污染

### 10.2 桌面壳层完整性

- 登录后可进入 Desktop Shell
- 可看到 Top Bar、Dock、桌面图标区
- 核心视觉风格与交互语言已统一

### 10.3 窗口系统完整性

- Dock 可打开或切换应用窗口
- 窗口支持聚焦、关闭、最小化、最大化
- 至少一个场景支持多窗口并行

### 10.4 业务主链路完整性

- Knowledge Explorer 可查看知识点 / 文档核心内容
- Training Console 可发起训练并进入作答主链路
- History Review 可查看训练历史与结果

### 10.5 质量门槛

- 桌面端主流分辨率下可用
- 移动端不会崩坏，至少进入降级版壳层
- 文档、脚本、运行说明已同步更新
