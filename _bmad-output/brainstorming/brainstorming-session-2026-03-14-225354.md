## Session Overview

**Topic:** InterviewOS V2 版本应该做哪些内容
**Goals:** 列出 V2 候选功能点；由用户最终拍板；同时明确本版本不做的内容；并标出每个点的优先级，控制开发周期

### Session Setup

本次脑暴聚焦于为 InterviewOS 的第二阶段确定合理版本边界。目标不是立即锁定单一方案，而是先尽可能发散出候选方向，再基于开发周期、业务价值和实现风险做优先级划分，并明确本版本不做的内容，避免范围失控。

## Technique Selection

**Approach:** Progressive Technique Flow
**Journey Design:** 先发散 V2 候选内容，再识别主题模式，然后从多视角审视高潜力方向，最后收敛为优先级与不做项。

**Progressive Techniques:**

- **Phase 1 - Exploration:** What If Scenarios
- **Phase 2 - Pattern Recognition:** Mind Mapping
- **Phase 3 - Development:** Six Thinking Hats
- **Phase 4 - Action Planning:** Decision Tree Mapping

**Journey Rationale:** 该流程适合版本规划场景，能够避免过早拍板，把自然想到的功能、真正有价值的方向、以及本阶段应主动放弃的内容清晰分层。

## Technique Execution Results

**What If Scenarios**

**[Category #1]**: 用户身份闭环
_Concept_: 为系统引入用户模块、登录态和数据隔离，让知识点、训练记录和历史结果不再是单机公开数据，而是归属于具体用户。这样 V2 才具备长期使用和跨设备连续训练的基础。  
_Novelty_: 它不是单纯“补一个登录页”，而是把整个系统从公开 MVP 变成真实可用产品。

**[Category #2]**: 批量知识导入
_Concept_: 用户不再手工一条条录入知识点，而是通过文件导入、批量粘贴或结构化导入，把已有资料更快转为训练素材。这样能显著降低首次使用门槛。  
_Novelty_: 核心不是“上传文件”本身，而是把知识输入效率提升为产品主能力。

**[Category #3]**: 标签化知识组织
_Concept_: 给知识点引入标签、分类或主题维度，让用户能够围绕岗位、语言、专题、难度组织训练范围。这样后续筛选、复习和推荐才有结构基础。  
_Novelty_: 它把知识点从平铺列表升级为可组合的训练资产。

**[Category #4]**: 模型响应性能优化
_Concept_: 优化与大模型交互时的反馈速度，包括 prompt 裁剪、缓存、超时控制、异步化或流式返回，让训练过程的等待时间下降。  
_Novelty_: 不是新增功能，而是直接修复“能不能持续用”的关键体验瓶颈。

**[Category #5]**: 前端体验升级
_Concept_: 对页面样式、信息层级和交互节奏做整体优化，让产品从“能用”提升到“愿意打开”。  
_Novelty_: 它关注的不是视觉修饰，而是把产品完成度从工程交付提升到用户感知层。

**[Category #6]**: 岗位/简历驱动的知识生成
_Concept_: 用户登录后先选择目标岗位，或上传自己的简历，由系统基于岗位要求和简历内容生成个性化知识点清单与训练方向。这样训练起点不再是用户手工定义，而是系统辅助建模。  
_Novelty_: 它把“知识录入”升级为“职业目标驱动的训练初始化”，更贴近真实面试准备场景。

**[Category #7]**: 今日推荐练习
_Concept_: 系统根据用户当前阶段、历史训练频率和知识掌握度，自动生成“今天该练什么”，降低选择成本。用户无需每次手动决定训练入口。  
_Novelty_: 从工具型产品向具备训练编排能力的学习产品迈进。

**[Category #8]**: 自动组题引擎
_Concept_: 基于标签、薄弱项、历史表现和目标岗位自动组题，动态组合训练内容，而不是始终由用户手工指定单一知识点。  
_Novelty_: 它引入的是“策略层”，让训练更像教练驱动而不是题库点选。

**[Category #9]**: 复习提醒与回练机制
_Concept_: 引入复习提醒、遗忘曲线式回练或阶段性回顾，让系统主动推动用户复训，而不是依赖用户想起来再回来练。  
_Novelty_: 它把 InterviewOS 从单次训练工具提升为持续复习系统。

**[Category #10]**: 面试场景定位锚点
_Concept_: 产品核心仍然应围绕面试准备、岗位匹配、模拟问答和复盘展开，而不是演化成泛化的知识管理或学习平台。V2 的所有候选能力都应服从这一定位。  
_Novelty_: 这不是一个功能点，而是一个版本筛选原则，用来判断哪些需求虽然“有用”，但会把产品带偏。

**[Category #11]**: V2 主线偏好
_Concept_: V2 更偏向“高频模拟面试训练”主线，辅以适度的个性化成长跟踪；岗位能力先保持轻量，只通过岗位选择来确定训练方向，不在本阶段深做复杂的岗位画像或差距分析。  
_Novelty_: 这给版本范围一个非常清晰的压缩原则，避免同时做成岗位平台、学习平台和教练系统。

**[Category #12]**: 首页直接暴露薄弱项
_Concept_: 用户打开系统后不先看全量知识点列表，而是优先看到当前最薄弱、最值得训练的知识点或主题。这样首页天然就是训练入口，而不是导航页。  
_Novelty_: 它把 Dashboard 从展示页变成行动页，直接服务高频训练。

**[Category #13]**: 一键推荐训练包
_Concept_: 系统基于薄弱项、标签和历史表现，自动推荐一个包含 3-5 题的训练包，用户一键进入，不需要自己挑知识点。  
_Novelty_: 把“开始训练”的成本压到最低，接近短时高频使用场景。

**[Category #14]**: 训练后自动生成薄弱项权重
_Concept_: 每次训练结束后，系统不仅给分，还要分析薄弱点、更新对应权重，并将结果反馈到后续推荐和回练策略中。  
_Novelty_: 这让评分不只是一次结果展示，而是直接驱动下一轮训练编排的输入。

**[Category #15]**: 题目缺乏真实面试感
_Concept_: 如果生成的问题太八股、太模板化、缺少对项目经历和真实场景的追问，用户会认为训练与真实面试脱节。V2 需要更强调场景化、项目化、岗位化的提问方式。  
_Novelty_: 关注的不是“题目数量”，而是题目的真实度和压迫感是否接近面试官。

**[Category #16]**: 评分可信度不足
_Concept_: 如果系统给出的评分和建议看起来空泛、机械或前后不一致，用户很快会失去对训练反馈的信任。V2 需要提高评分解释性和稳定性。  
_Novelty_: 评分不只是算法输出，它是整个产品可信度的核心。

**[Category #17]**: 训练负担过重
_Concept_: 如果问题连续过难、没有提示、用户中途频繁卡住，训练就会从“高频练习”变成“高压消耗”。可以考虑为题目增加提示项、追问层级或渐进式难度。  
_Novelty_: 它把训练体验从纯评测转向“可持续完成”的设计。

**[Category #18]**: 响应等待破坏节奏
_Concept_: 由于强依赖大模型生成题目和评分，如果等待时间过长，用户会在一次短时训练中频繁断节奏，影响留存。V2 需要把性能优化视为主链路能力。  
_Novelty_: 这不是单纯后端优化，而是直接影响产品节奏感的关键体验问题。

**[Category #19]**: 进步感不明显
_Concept_: 如果用户训练后看不到掌握度变化、薄弱项改善或阶段成果，就难以建立长期坚持的动力。V2 需要让进步被可视化且能被理解。  
_Novelty_: 它把“训练结果”转化为“继续训练的动机”。

**[Category #20]**: 暂不做简历深度解析
_Concept_: V2 可以只保留岗位选择或非常轻量的简历入口，不深做复杂的简历结构化解析、项目经历抽取和差距建模。  
_Novelty_: 主动压缩范围，避免一开始把产品做成职业画像系统。

**[Category #21]**: 暂不做复杂岗位画像
_Concept_: 不在 V2 建立复杂的岗位能力模型、岗位级胜任力图谱或岗位差距分析引擎，只做方向级岗位选择即可。  
_Novelty_: 保证岗位能力服务训练，不反过来吞掉训练主线。

**[Category #22]**: 暂不做多轮对话
_Concept_: V2 不扩展为持续对话式面试官，也不做复杂追问链路管理，先把单轮或短链路训练体验做扎实。  
_Novelty_: 防止上下文管理、会话状态和成本控制过早失控。

**[Category #23]**: 暂不做复杂图表统计
_Concept_: V2 不追求复杂成长报表、雷达图和多维分析大屏，只保留能支撑进步感的最小可视化。  
_Novelty_: 让进步展示服务训练，而不是演化为 BI 页面。

**[Category #24]**: 暂不做社交/排行榜
_Concept_: V2 不引入社交互动、排行榜、多人比较等机制，聚焦个人面试训练闭环。  
_Novelty_: 明确产品不是社区，也不是竞赛平台。

**[Category #25]**: 暂不做过多文件格式导入
_Concept_: 文件导入若进入 V2，也只应支持少量高价值格式，不扩展到过多复杂格式和解析链路。  
_Novelty_: 将知识导入控制在“够用”，不演化成内容处理平台。

**[Category #26]**: 暂不做超复杂标签系统
_Concept_: 标签体系可以有，但不应在 V2 做成多层级、多维度、重配置的复杂分类系统。  
_Novelty_: 标签服务训练推荐即可，不做复杂知识治理。

**[Category #27]**: 暂不做双数据库拆分
_Concept_: 继续保持 PostgreSQL 单库主路径，不在 V2 推进 MySQL + PostgreSQL 双库职责拆分。  
_Novelty_: 避免基础设施复杂度先于业务价值增长。

**[Category #28]**: 暂不做复杂权限体系
_Concept_: V2 只需满足单用户登录和数据隔离，不引入组织级、角色级、多租户权限体系。  
_Novelty_: 让鉴权只解决当前安全边界问题，不做平台化设计。

**[Category #29]**: 暂不做语义检索 / RAG
_Concept_: V2 不把重点放在向量检索、RAG 或复杂知识召回链路上，先验证高频训练闭环是否成立。  
_Novelty_: 避免 AI 基础设施炫技先于真实训练体验闭环。

**[Category #30]**: 题目类型分层
_Concept_: 训练题目应明确区分八股题、项目题、实际场景题等不同类型，让训练内容更接近真实面试中的问题结构，而不是所有题都使用一种模板。  
_Novelty_: 它提升的不是题目数量，而是题型分布的真实性。

**[Category #31]**: 难度梯度设计
_Concept_: 题目需要具备从基础到深入的难度梯度，避免训练始终过浅或过重，并支持用户逐步进入状态。  
_Novelty_: 它把训练从静态问答升级为可控强度的练习过程。

**[Category #32]**: 避免重复问法
_Concept_: 系统应减少题目表达和问法重复，避免用户产生“系统一直在换皮重复问”的疲劳感。  
_Novelty_: 关注的是训练新鲜感与有效性，而不是单题质量本身。

**[Category #33]**: 追问式问题链
_Concept_: 问题应尽量具备真实面试中的延展感，能从一个回答继续衍生出下一个问题，而不是每道题彼此完全割裂。  
_Novelty_: 这保留了部分真实面试的压力和连续思考特征，即使 V2 不做完整多轮对话。

**[Category #34]**: 提示但不直接给答案
_Concept_: 当用户卡住时，系统可以提供思考维度、关键词或提示项，而不是直接给标准答案，帮助用户继续回答。  
_Novelty_: 它让训练体验兼顾可完成性与思考价值，避免沦为填鸭式提示。

**[Category #35]**: 评分结果要可理解
_Concept_: 当前“准确度、深度、清晰度、优点、建议”这类维度虽然完整，但用户阅读后容易不知所云。V2 可以把分数转化为更直观的档位解释，例如“表达不清晰”“回答不完整”“有基础但不够深入”等。  
_Novelty_: 从抽象评分维度转向用户能立即理解的反馈语言。

**[Category #36]**: 扣分原因要具体
_Concept_: 用户不仅需要知道得了多少分，还需要知道具体扣在什么地方，例如概念错误、缺少关键点、逻辑跳跃、表达混乱。  
_Novelty_: 它让评分具备可操作性，而不是停留在结果展示。

**[Category #37]**: 建议答案要贴近真实表达
_Concept_: AI 给出的参考答案不应过度标准化、书面化或像教科书，而应更接近真实候选人在面试中能说出的表达方式和回答结构。  
_Novelty_: 它提升的是答案示范的可模仿性，而不是内容完整度本身。

**[Category #38]**: 评分结果页先给档位与主要问题
_Concept_: 用户进入结果页后，首先应看到清晰的档位判断和简洁的主要问题摘要，而不是长篇维度说明。档位定义需要细化到用户能够直接感知自己属于“表达问题”“内容缺失”还是“深度不足”等类别。  
_Novelty_: 它把结果页从分析报告改成可立即读懂的训练反馈面板。

**[Category #39]**: 中段展示缺失点与更好回答思路
_Concept_: 在结果页第二层展示回答缺失了哪些关键点，以及如果重答，应该沿什么思路组织答案。重点是告诉用户“下一次怎么答会更好”。  
_Novelty_: 反馈从静态评分变成可直接用于下一次训练的指导。

**[Category #40]**: 结尾提供自然参考表达
_Concept_: 结果页最后再给出一个更自然、更贴近真实面试表达的参考回答，而不是僵硬的标准答案。  
_Novelty_: 它强化的是模仿价值和表达迁移能力，而不是背诵价值。

## Idea Organization and Prioritization

**Thematic Organization:**

**Theme 1: 高频训练主链路**
_Focus_: 缩短用户从打开系统到完成一次训练的路径，让系统主动推动训练发生。_

- 首页直接暴露薄弱项
- 一键推荐 3-5 题训练包
- 自动组题引擎
- 复习提醒与回练机制
- 训练后自动更新薄弱项权重

**Theme 2: 题目真实性与训练质量**
_Focus_: 让训练题不再只是“AI 出题”，而更接近真实面试官的提问方式。_

- 题目类型分层
- 难度梯度设计
- 避免重复问法
- 追问式问题链
- 提示但不直接给答案
- 题目缺乏真实面试感（风险约束）

**Theme 3: 评分可信度与结果反馈**
_Focus_: 让用户相信评分结果，并知道下一次如何回答得更好。_

- 评分结果要可理解
- 扣分原因要具体
- 建议答案要贴近真实表达
- 评分结果页先给档位与主要问题
- 中段展示缺失点与更好回答思路
- 结尾提供自然参考表达
- 评分可信度不足（风险约束）
- 进步感不明显（风险约束）

**Theme 4: 用户进入门槛与训练方向**
_Focus_: 降低用户开始使用系统的成本，并让训练方向更贴近面试目标。_

- 用户身份闭环
- 岗位/简历驱动的知识生成
- 批量知识导入
- 标签化知识组织
- V2 主线偏好
- 面试场景定位锚点

**Theme 5: 体验基础设施**
_Focus_: 保证训练过程顺畅、轻量、可持续，而不是卡顿或过载。_

- 模型响应性能优化
- 前端体验升级
- 训练负担过重（风险约束）
- 响应等待破坏节奏（风险约束）

**Non-Goals / 明确不做项**

- 暂不做简历深度解析
- 暂不做复杂岗位画像
- 暂不做多轮对话
- 暂不做复杂图表统计
- 暂不做社交/排行榜
- 暂不做过多文件格式导入
- 暂不做超复杂标签系统
- 暂不做双数据库拆分
- 暂不做复杂权限体系
- 暂不做语义检索 / RAG

**Prioritization Results:**

- **Top Priority Themes:** Theme 4 用户进入门槛与训练方向、Theme 2 题目真实性与训练质量、Theme 3 评分可信度与结果反馈
- **Quick Win Themes:** Theme 5 体验基础设施、Theme 4 用户进入门槛与训练方向
- **Breakthrough Themes:** Theme 1 高频训练主链路、Theme 2 题目真实性与训练质量、Theme 3 评分可信度与结果反馈

**Priority Insight:**
用户当前更倾向先把“用户如何进入系统、系统给出什么题、系统如何反馈结果”三件事做扎实；而真正能拉开产品差异的，是后续把这些能力串成高频训练闭环。也就是说，V2 的实施顺序与长期差异化方向并不完全相同。

**Action Planning:**

**建议的 V2 版本目标**

- V2 聚焦高频面试训练体验升级，优先补齐用户进入、题目真实性与评分可信度。

**建议的优先级分层**

- **P0：版本骨架**
  - 用户模块
  - 岗位选择确定训练方向
  - 批量导入知识点
  - 简单标签体系
  - 题目类型分层
  - 难度梯度
  - 避免重复问法
  - 评分档位化
  - 主要问题摘要
  - 缺失点与更好回答思路
  - 更自然的参考表达

- **P1：体验增强**
  - 模型响应性能优化
  - 页面样式和训练流程优化
  - 提示但不直接喂答案
  - 训练结果可视化进步感

- **P2：差异化增强**
  - 首页直接暴露薄弱项
  - 一键推荐 3-5 题训练包
  - 自动组题
  - 训练后更新薄弱项权重
  - 回练提醒

**明确不做项**

- 简历深度解析
- 复杂岗位画像
- 多轮对话
- 复杂图表统计
- 社交/排行榜
- 过多文件格式导入
- 超复杂标签系统
- 双数据库拆分
- 复杂权限体系
- 语义检索 / RAG

## Session Summary and Insights

**Key Achievements:**

- 明确了 V2 的产品主线是高频模拟训练，而不是泛知识管理平台。
- 将候选需求收敛为 5 个主题簇，并区分了高优先级、Quick Wins 与 Breakthrough 方向。
- 明确了题目真实性与评分可信度是决定 V2 成败的核心体验约束。
- 形成了清晰的不做项列表，能够有效控制版本范围。

**Session Reflections:**

本次脑暴最重要的突破，不是新增了多少功能点，而是建立了一个稳定的版本筛选原则：所有能力都必须服务于“更快进入训练、更像真实面试、更容易理解反馈、更愿意持续使用”。在此基础上，V2 可以先完成可用骨架，再逐步演进到真正的高频训练闭环。
